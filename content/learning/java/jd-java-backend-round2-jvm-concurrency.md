## 2、JVM内存区域划分，哪些区域会发生OOM，场景举例

JVM内存分为**堆、虚拟机栈、本地方法栈、方法区（元空间）、程序计数器**。

除程序计数器外，其余区域均可发生OOM：
- **堆OOM**：常见于无限创建对象、内存泄漏
- **栈OOM**：见于递归过深、线程过多
- **元空间OOM**：见于大量动态生成类、代理类

---

## 3、GC垃圾回收机制，G1和ZGC核心区别、适用场景

GC核心是识别无效对象、回收空闲内存。

**G1**：采用分代回收、区域化内存管理，兼顾吞吐量和低延迟，适合中端服务

**ZGC**：低延迟垃圾收集器，基于染色指针、内存屏障，停顿时间极低（毫秒级），适合高并发、低延迟的线上核心服务。京东线上核心业务多采用ZGC。

---

## 4、哪些对象会进入老年代，对象晋升机制

- 新生代对象经历 **15次Minor GC** 未回收 → 进入老年代
- 新生代**大对象**直接分配到老年代
- Minor GC后存活对象过多、剩余空间不足 → 触发对象**提前晋升**

---

## 5、线上OOM问题如何排查，完整调优流程

1. **定位**：通过日志、监控定位OOM类型
2. **Dump**：导出堆dump文件，借助**MAT工具**分析大对象、内存泄漏点
3. **排查代码**：内存泄漏、无限创建对象
4. **调参**：调整JVM参数（堆内存、GC收集器）
5. **验证**：压测验证优化效果，持续监控线上指标

> 辅助工具：**JFR、JMC** 可实时监控JVM运行状态

## 7、synchronized和Lock锁底层区别，性能对比

> ⚠️ 原文此处被XHS 1000字限制截断，以下为补充完善：

### synchronized（JVM内置锁）
- **实现**：JVM层面实现，monitorenter/monitorexit字节码指令
- **锁升级**：无锁 → 偏向锁 → 轻量级锁(CAS + 自旋) → 重量级锁(Monitor)
- **释放**：自动释放（退出同步块或异常）
- **不可中断**：等待获取锁时不响应interrupt
- **非公平锁**

### Lock（如ReentrantLock）
- **实现**：Java API层面，基于AQS（AbstractQueuedSynchronizer）
- **灵活性**：支持公平/非公平锁、可中断等待（lockInterruptibly）、超时获取（tryLock）、多条件变量（Condition）
- **释放**：必须手动unlock()，建议在finally中释放
- **性能**：JDK 6+ synchronized经过锁升级优化后，两者性能差距已不大；简单同步场景推荐synchronized，复杂场景（需要中断、超时、多条件）推荐Lock
