## 面试总结（13题）

### 1、自我介绍
有点短了，尽量三分钟，只说了自己叫什么，什么学校，什么专业，什么公司，做的什么内容，可以加一部分职业的规划。

**补全答案**：
自我介绍可以按“背景 -> 项目 -> 技术栈 -> 亮点 -> 求职方向”来讲，控制在 2 到 3 分钟。

示例结构：

1. 我是 Java 后端方向，主要做过 Spring Boot / Spring Cloud 体系下的业务系统开发。
2. 最近项目里负责过优惠券、购物卡、订单相关模块，涉及 Redis 缓存、RabbitMQ 异步解耦、Redisson 分布式锁、线程池异步任务等。
3. 我比较熟悉高并发场景下的库存扣减、幂等、防重复消费、缓存一致性和接口性能优化。
4. 后续希望继续做偏业务复杂度和稳定性要求较高的后端开发，把并发、缓存、消息队列这些能力用扎实。

### 2、你们的项目里用到了什么组件
这个要背一下，只说到了 Nacos、OpenFeign、Redis、RabbitMQ。

**补全答案**：
可以按组件职责来答，而不是只报名字：

- Nacos：服务注册发现和配置中心，服务启动后注册实例，调用方通过服务名发现目标服务。
- OpenFeign：声明式 HTTP 调用，配合负载均衡和熔断降级，减少手写远程调用代码。
- Redis：做热点数据缓存、库存预扣、分布式锁、计数器、唯一 ID 辅助生成等。
- RabbitMQ：做异步解耦、削峰填谷、延迟处理，例如优惠券发放后异步生成购物卡或发送通知。
- Redisson：封装分布式锁，解决并发领取、库存扣减、重复提交等临界区问题。
- 线程池：承接耗时任务和批量任务，避免请求线程长时间阻塞。

### 3、rabbitmq怎么保证消息不丢失
这个可以。

**补全答案**：
RabbitMQ 防消息丢失要从生产者、Broker、消费者三段说：

- 生产者：开启 publisher confirm，确认消息是否到达交换机；开启 return callback 或 mandatory，处理路由不到队列的消息。
- Broker：队列、交换机和消息都持久化；重要消息设置 deliveryMode=2；集群场景可用镜像队列或 quorum queue 提升可靠性。
- 消费者：关闭自动 ack，业务处理成功后手动 ack；失败时根据异常类型决定 requeue、重试、死信队列。
- 业务兜底：本地消息表或 outbox 记录待发送消息，定时补偿；消费端幂等，防止重投导致副作用。

一句话版：发送确认、消息持久化、手动 ack、失败重试和业务补偿一起保证可靠投递。

### 4、怎么保证消息不重复消费
这个可以。

**补全答案**：
消息队列通常只能保证“至少一次投递”，重复消费要靠业务幂等：

- 给每条业务消息设计唯一业务键，例如 orderId、couponId + userId、requestId。
- 消费前先查幂等表或 Redis setnx，已处理直接 ack。
- 数据库层加唯一索引，例如用户领取优惠券表对 `user_id + coupon_id` 建唯一约束。
- 对状态流转类业务，用状态机限制重复执行，例如只有 `INIT -> SUCCESS` 能更新。
- 消费失败要区分“业务已落库但 ack 失败”和“业务未执行成功”，避免盲目重试。

面试里可以补一句：防重复消费不是只靠 MQ 配置，最终要落到消费端幂等设计。

### 5、redis是怎么用的
答了缓存，还有 Redis + Lua 脚本的优惠券库存校验。

**补全答案**：
项目里 Redis 可以从三个层次讲：

- 缓存：缓存商品、优惠券活动、用户权益等读多写少数据，用 Cache Aside 模式，先查缓存，未命中查 DB 后回填。
- 原子扣减：优惠券库存校验、用户是否领取过、库存扣减可以放在 Lua 脚本里一次执行，避免并发下先查后扣的不一致。
- 分布式协作：用 Redisson 分布式锁保护复杂临界区，用 Redis 自增或号段方案辅助生成业务编号。

注意补上风险：缓存要考虑穿透、击穿、雪崩和一致性；Lua 适合短逻辑原子操作，不适合放复杂业务。

### 6、怎么实现的redisson分布式锁？
答了自己进行的二次封装，主要就是一个 AOP 加自定义注解加 SpEL 表达式，用了工厂模式选择锁的类型，用了策略模式选择失败重试策略，用 SpEL 表达式动态获取锁的名称。

**补全答案**：
可以这样讲实现链路：

1. 定义 `@DistributedLock` 注解，包含锁 key、锁类型、等待时间、租约时间、失败策略等参数。
2. 用 AOP 拦截标注方法，解析方法参数和 SpEL，生成业务锁 key，例如 `coupon:lock:#{userId}:#{couponId}`。
3. 根据锁类型选择 `RLock`、`RReadWriteLock`、`RFairLock` 等实现。
4. 调用 `tryLock(waitTime, leaseTime)` 获取锁，获取失败按策略处理，例如快速失败、重试、降级。
5. 用 `try-finally` 保证释放锁，释放前判断当前线程是否持有锁，避免误删别人的锁。

可以强调：封装的价值不是“造锁”，而是统一锁命名、超时策略、失败策略和监控日志。

### 7、如果说redisson获取到锁了，但是挂了，没有释放锁怎么办（没有答出来）
没有答出来，懵了，应该答 Redis 可以设置过期时间，后续可以答 Redisson 的看门狗机制。

**补全答案**：
Redisson 获取锁后会在 Redis 写入带过期时间的 key，所以服务宕机后锁不会永久存在。默认如果没有显式传 `leaseTime`，Redisson 会启用 watchdog 看门狗机制：持锁线程还活着时，后台定时给锁续期；如果 JVM 宕机或线程没了，续期停止，锁到期自动释放。

答题要点：

- 显式传 `leaseTime`：锁到期自动释放，不依赖看门狗，但业务执行超过租约可能提前释放。
- 不传 `leaseTime`：Redisson 默认看门狗续期，适合执行时间不确定的任务。
- 最佳实践：锁粒度要小，业务耗时要可控，释放锁要放 `finally`，释放前判断 `isHeldByCurrentThread()`。

也可以补充局限：Redis 分布式锁不是强一致事务锁，极端网络分区、主从切换场景仍要结合业务幂等兜底。

### 8、你们有用线程池吗
说到了优惠券。

**补全答案**：
可以说在线程池主要用于“非核心链路、耗时任务、批量任务”：

- 优惠券或购物卡批量生成：主流程先落任务状态，具体生成异步执行。
- 消息消费后的异步处理：例如发通知、写操作日志、统计埋点。
- 批量 IO：调用外部系统或批量查询时控制并发，避免请求线程阻塞太久。

同时要说明线程池参数不是随便配：核心线程数、最大线程数、队列长度、拒绝策略都要结合任务是 CPU 密集还是 IO 密集来定。

### 9、你们是用原生的线程池还是自己写的
说的是自己写的。

**补全答案**：
更准确的表达应该是“基于 JDK `ThreadPoolExecutor` 做了业务封装”，不是重新实现线程池。

封装内容可以包括：

- 统一线程名前缀，方便日志和问题定位。
- 参数可配置，例如核心线程数、最大线程数、队列大小。
- 统一拒绝策略，例如告警、降级、落库等待补偿。
- 包装任务上下文，例如 traceId、MDC、用户上下文传递。
- 暴露监控指标，例如 activeCount、queueSize、completedTaskCount。

这样答比“自己写线程池”更稳，因为面试官可能会追问你是否真的实现了调度、队列、worker 生命周期。

### 10、为什么不用原生的线程池？
答得不好，说的是有参数是不可以修改的

**补全答案**：
这里不要说“不用原生线程池”，应该说“不直接使用 `Executors` 的快捷工厂方法，而是显式创建 `ThreadPoolExecutor`”。

原因：

- `FixedThreadPool` 和 `SingleThreadExecutor` 默认使用无界队列，请求堆积时可能 OOM。
- `CachedThreadPool` 默认最大线程数接近无限，大量请求时可能创建过多线程导致 CPU 抖动甚至宕机。
- `ScheduledThreadPool` 也要注意任务堆积和异常中断问题。
- 快捷工厂方法隐藏了队列、线程数、拒绝策略等关键参数，不利于容量评估和线上治理。

更好的答法：

```java
new ThreadPoolExecutor(
    corePoolSize,
    maximumPoolSize,
    keepAliveTime,
    TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(queueSize),
    namedThreadFactory,
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

然后结合监控动态调整或配置化管理，而不是运行时随意修改参数。

### 11、为什么购物卡要用线程池？
我说自己的理解是减少了io，实际上是不对的

**补全答案**：
购物卡批量生成通常有大量记录生成、写库、状态更新、可能还包含加密/编码/导出等耗时操作。如果放在请求线程里同步执行，接口会变慢，也容易超时。

使用线程池的价值：

- 异步化：用户请求只提交生成任务，立即返回任务受理结果。
- 控并发：限制同时生成的任务数量，保护数据库和下游系统。
- 削峰：高峰期任务进入队列，后台按能力消费。
- 可观测：任务状态可记录为待执行、执行中、成功、失败，失败后可以重试。

注意不要只说“减少 IO”。更准确是：通过异步和并发控制缩短用户等待时间，并保护系统资源。

### 12、购物卡为什么要防止刷爆
答的用了 Redis 的自增 ID，它有规律性，容易被人刷。

**补全答案**：
购物卡本质上是有价值凭证，如果卡号、兑换码或领取接口可被枚举，就可能被恶意脚本撞库、批量领取、批量兑换。

防刷要从几层做：

- 编码不可预测：不要直接暴露自增 ID；兑换码使用足够长的随机数、签名或加密编码。
- 接口限流：按用户、IP、设备、活动维度限流。
- 幂等和唯一约束：同一用户同一活动只能领取一次，数据库唯一索引兜底。
- 风控校验：异常频率、异常 IP、异常设备进入黑名单或验证码。
- 库存原子扣减：Redis Lua 或数据库条件更新防止超发。
- 核销校验：兑换时校验状态、有效期、绑定用户、是否已使用。

一句话版：防刷爆不是只防 ID 枚举，而是保护“库存、领取资格、兑换码、核销状态”这一整条链路。

## 复习重点

这篇复盘最值得补强的是：

1. RabbitMQ 可靠投递：confirm、return、持久化、手动 ack、死信和补偿。
2. 消费幂等：唯一业务键、唯一索引、状态机、Redis setnx。
3. Redisson 锁：过期时间、watchdog、`try-finally`、`isHeldByCurrentThread()`。
4. 线程池：不要直接用 `Executors`，要显式配置队列、线程数、拒绝策略和监控。
5. 购物卡防刷：随机码、限流、幂等、风控、库存原子扣减、核销状态校验。
