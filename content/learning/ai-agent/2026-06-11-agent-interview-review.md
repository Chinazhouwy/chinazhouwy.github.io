## 背景

持续更新训练营同学面经，以下是同学面试经验脱敏整理后内容。

主要问：东西跑起来没有，有没有人真在用，你清不清楚它的边界在哪。

---

## 一、Agent项目被一层层问

有一场顺着我简历上的一个 agent 项目往下挖，几乎是一层层剥。先让我讲整体架构和呈现形式，再追问 agent 之间怎么协同，有没有什么特殊设计；讲到状态怎么维护，他直接问是不是 **LangGraph** 写的。

中间有一道反复确认的，是这个系统的**输入到底是什么**。我讲过一遍，他没完全理解，又让我重说一次。这说明我没讲好，描述一个 agent 系统，应该先把输入是什么，输出是什么，解决什么问题讲清楚，一句话让对方能复述出来，再去铺架构。

### 最难的一道对比题

同一场最后一道方案对比，是几场里最难接的。

他先把我项目概括了一遍，说白了你这是个在代码里找并发错误的 agent，被分析的代码大多是底层那类语言写的。确认之后他抛出来：现在成的 coding agent 已经很强了，自带大量工具调用，那我完全可以走另一条路——直接用现成的 coding agent，配一个外部知识库，再把你们那些私有的业务写法 skill 的形式扩进去，让它去，去验证并发错误。这是方案 B，你这套是方案 A，A 比 B 好在哪。

我当时答得一般，没说到点上。后来才想清楚，我真正能站住的地方，不在"我把模型调得更巧"——底层模型还在往前走，编排层那点优势迟早会被抹平。要是 A 和 B 的差别说到底只剩一层 prompt 和一滩胶水代码，那 B 会赢。能撑住的跟模型推理无关的那部分：**把代码跑起来，控制调度**。

---

## 二、上下文工程

后面好几场也在反复绕 context engineering 这个词。

一个任务要经过好几个 agent，它们之间来回传递，上下文会越积越长，而窗口是有上限的。我的处理是，agent 交接的时候别把那份完整的历史上下文整个甩给下一个，只传一份摘要或者那个最要用到的那个产物；每个 agent 的上下文按自己的职责截到最小；像完整源码这种大块内容，放到外部存储或检索里，传个引用过去。
