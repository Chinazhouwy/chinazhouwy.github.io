## 参考答案摘编

### Q8: 如何实现仅仅毫秒级别的延迟？

**答题思路**：

毫秒级延迟的核心在于 **端到端的链路优化**，不单是某一个环节的优化：

1. **模型侧**：选择低延迟推理引擎（vLLM/TensorRT-LLM），使用 FlashAttention 优化，量化（INT4/INT8）减少显存带宽占用，Prefill + Decode 分离部署
2. **网络侧**：就近部署、CDN加速，SSE/WebSocket 长连接复用，避免每次新建连接
3. **计算侧**：预计算热点结果（缓存），非核心路径异步化，减少同步等待
4. **工程侧**：连接池复用、序列化优化（Protobuf > JSON）、减少不必要 I/O

> ⚠️ 工程踩坑：很多时候延迟并非模型慢，而是服务链路中某个环节的线程阻塞或 GC 停顿导致的，先用 Arthas 或 JFR 定位实际瓶颈再优化

### Q18: 系统支持多轮对话吗？如果上下文爆炸或工具循环调用，你是怎么解决的？

**答题思路**：

多轮对话的上下文管理是 Agent 系统的核心挑战之一：

**上下文爆炸解决方案**：

1. **滑动窗口裁剪**：保留最近的 N 轮对话，历史对话做摘要压缩后追加到系统 prompt
2. **分层记忆**：
   - 短期记忆：当前 session 的完整对话（保留最近 K 轮）
   - 长期记忆：重要的信息存储到外部存储（PostgreSQL/Redis），通过检索召回到上下文
3. **工具调用降维**：限制递归深度，设置最大工具调用轮次上限（如 10 轮），超过后强制输出

**工具循环调用解决方案**：

1. **循环检测**：记录已调用的工具和参数 hash，检测到重复调用相同工具+相同参数时强制中断
2. **最大深度限制**：设置 max_iterations
3. **退化策略**：循环超过阈值后降级为单次回复模式

> ⚠️ 工程踩坑：上下文窗口不是越大越好——长上下文会让模型"注意力涣散"，回复质量下降。比起无脑扩窗口，更好的做法是精准召回 + 摘要压缩

### Q28: RAG项目二中MySQL和ES数据一致性是怎么做的

**答题思路**：

MySQL（主库）与 ES（搜索引擎）之间的数据一致性是经典的双写一致性问题：

**方案一：Canal + MQ 异步同步**
- MySQL 开启 binlog（ROW 模式）
- Canal 监听 binlog 变更事件
- 将变更事件投递到 MQ（RocketMQ/Kafka）
- 消费端写入 ES
- 优点：对业务代码无侵入，解耦
- 缺点：存在秒级延迟，Canal 服务需要高可用

**方案二：业务双写 + 本地消息表**
- 业务代码写 MySQL 的同时写本地消息表
- 定时任务扫描消息表，异步同步到 ES
- 消费成功后删除消息记录
- 优点：精确控制，事务保障
- 缺点：代码侵入，需要维护消息表

**核对机制**：
- 定时任务对比 MySQL 和 ES 的记录数 + 更新时间
- 发现不一致时触发全量或增量修复

> ⚠️ 工程踩坑：ES 的 refresh_interval 默认 1s，写入后立即查询可能查不到，需要根据业务容忍度调整。数据一致性链路中要加监控告警，延迟超过阈值立即告警

### Q33: sse流式输出在用户关闭浏览器的时候就断开了，如何保证中断后还能保持之前输出

**答题思路**：

SSE 断连恢复的核心是 **异步任务执行 + 断点续传**：

1. **异步任务化**：用户发起请求时，后台创建异步任务（生成唯一 taskId），任务独立于 HTTP 连接持续执行
2. **中间结果持久化**：每生成一段 token/步骤，写入 PostgreSQL 或 Redis，记录当前进度 checkpoint
3. **断线重连**：
   - 前端监听 `onerror`，检测到连接断开后自动重连
   - 重连时带上上次的 cursor/taskId
   - 后端从 checkpoint 恢复，从断点处继续推送
4. **兜底轮询**：如果在重试 N 次后 still 无法建立 SSE，降级为轮询模式

> ⚠️ 工程踩坑：断连恢复时要防止重复生成——用乐观锁或版本号确保每个 chunk 只生成一次。同时考虑服务重启场景：task 的中间状态需要持久化，不能只存在内存里

### Q47: Checkpoint 存储为什么选择 PostgreSQL JSONB，而不使用 Redis 或分表存储？

**答题思路**：

选择 PostgreSQL JSONB 而非 Redis 的原因：

1. **持久化与可靠性**：Agent 的 checkpoint 是业务关键状态，丢失可能导致数据不一致，Redis 作为缓存层存在丢数据风险
2. **查询灵活性**：JSONB 支持任意字段的索引查询和部分更新（`jsonb_set`），适合存储半结构化的 Agent 状态
3. **事务支持**：PostgreSQL 的 ACID 特性保证并发场景下的状态一致性，Redis 的事务较弱
4. **存储成本**：checkpoint 数据可能很大（包含完整对话上下文），Redis 内存成本远高于 PG 磁盘存储
5. **数据关联**：checkpoint 通常与用户、session 等关系型数据关联查询，JSONB 可以 JOIN 操作

**何时该用 Redis**：
- 需要极低延迟的临时状态缓存
- 状态数据可以接受丢失
- 状态结构固定不需要复杂查询

> ⚠️ 工程踩坑：JSONB 不是银弹，频繁的大字段更新会导致 PG 的 WAL 膨胀和 VACUUM 压力，建议定期归档旧 checkpoint 到冷存储

### Q49: 多模型选型中，为何选择 DeepSeek AI？具体的选型标准是什么？

**选型标准**：

1. **性能指标**：MMLU、HumanEval、GSM8K 等基准测试评分，DeepSeek 在同等参数规模下表现领先
2. **推理成本**：DeepSeek 开源模型可私有化部署，token 成本远低于闭源 API，高并发场景优势明显
3. **上下文长度**：128K/1M 上下文窗口，适合 Agent 场景的长对话和长文档理解
4. **中文能力**：DeepSeek 在中文理解和生成质量上对标 GPT-4
5. **工具调用**：Function Calling / Tool Use 能力成熟，原生支持 Agent 场景
6. **可定制性**：开源权重支持 LoRA 微调，可以针对业务场景做领域适配

> ⚠️ 工程踩坑：模型选型不能只看 benchmark，还要测试 API 稳定性、并发 latency P99、服务可用性 SLA，以及是否支持批量推理和流式输出等工程特性
