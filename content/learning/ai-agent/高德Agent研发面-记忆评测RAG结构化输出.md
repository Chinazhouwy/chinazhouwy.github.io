## 一、Agent 记忆写入机制设计

### 核心问题

**如何设计 Agent 的记忆写入机制？面对海量对话，依据什么标准筛选"写入内容"？**

### 记忆筛选标准

| 维度 | 说明 | 示例 |
|------|------|------|
| **持久性价值** | 长期保存的事实性信息 | 会议时间、用户偏好 |
| **结构化程度** | 高结构化：实体与关系 | Entity & Relation |
| **个性化价值** | 用户画像：习惯、偏好 | 习惯、偏好、编号 |
| **隐藏需求** | 表面话语背后的意图 | Intent |

### 混合存储架构

```
┌─────────────────────────────────────────────────┐
│              混合存储架构 (Hybrid Storage)         │
├─────────────────────────────────────────────────┤
│  向量数据库      图数据库      关系型DB           │
│  (Vector DB)    (Graph DB)   (RDBMS)            │
├─────────────────────────────────────────────────┤
│  存储: Event Log        Core Profile            │
└─────────────────────────────────────────────────┘
```

### 动态维护机制

- **评分系统**：重要性加权（如金钱/合同高权重）
- **反思 (Reflection)**：定期总结深层洞察
- **专家 (Expert)**：根据数据结构类型，分流存储

## 二、Agent 评估体系

### 三层全景架构

#### 第一层：核心量化指标 (Metrics Dimension)

| 维度 | 指标 | 说明 |
|------|------|------|
| **结果维度 (Task-Level)** | 任务成功率 (Task Success Rate) | 最终任务是否完成 |
| | 环境终态匹配度 (State Matching) | 结果状态是否符合预期 |
| **过程维度 (Trajectory)** | 工具调用准确率 (API Precision) | 调用工具是否正确 |
| | 规划效率 | 步数 vs 最优路径 |
| | 自我纠错率 (Self Refine Rate) | 能否自动修正错误 |
| **系统维度 (System-Level)** | 端到端延迟 (Latency) | 响应时间 |
| | Token 消耗量与 API 成本 | 成本控制 |
| | 平均交互轮数 | 效率 |

## 三、RAG Query 改写

### 面试问题

**面试官：你的 RAG 系统是怎么进行 Query 改写的？这可不是三个简单的算法题**

### Query 改写架构与实战

#### 核心挑战

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **响应延迟 (Latency)** | 改写增加 LLM 调用，导致 TTFT 变长 | 使用 SLM (小模型) 专项改写 + 并行化检索流水线 |
| **查询漂移 (Drift)** | 改写后偏离原意，导致召回无关内容 | 嵌入语义相似度阈值过滤 + 保留原始 Query 混合检索 |
| **成本开销 (Cost)** | 每条请求多消耗 Token | 前置意图路由分类，仅对复杂 Query 开启高阶改写 |

## 四、大模型结构化输出

### 溯源：为什么 LLM 难以稳定输出 JSON？

| 问题 | 说明 |
|------|------|
| **Next Token 机制** | 本质是概率预测，没有显式的语法状态机 |
| **对话式对齐** | RLHF 习惯性导致模型输出解释性"废话"前缀 |
| **Token 截断冲突** | 特殊字符在大语言模型编码中具有复杂的权重分布 |

### 专家级核心策略

#### 1. 防御性编程

> 永远不要信任模型的原始输出，必须通过校验网关。

#### 2. 方案分级

> 小打小闹用 Prompt，正儿八经做业务，必须上 Tool Calling 甚至约束解码。

#### 3. 反馈闭环

> 错并不可怕，可怕的是你没有自动纠错的机制。

### 核心原则

> **用确定性的架构去约束概率性的输出，这就是专家。**
