## 背景

大家好，我是 Agent 搬砖日记。从 23 年开始做 Agent 到现在，见证了这个赛道从 Demo 满天飞到如今深水区落地的全过程。最近和阿里淘天某核心业务线的技术负责人聊了一轮。整体感觉是：**大厂现在对 Agent 的考察，已经完全脱离了"调包侠"阶段，非常看重工程化落地能力和模型迭代闭环**。

今天复盘一下这一面中比较有质量的几个探讨点（隐去了具体项目细节，只聊技术逻辑）：

---

## 一、RAG 架构：不仅要准，还要快

### Q：关于 RAG 中的 Chunking 策略与向量库选型逻辑？

**✅ 应对思路：**

**分块策略**：我强调了不能迷信固定的 Token 长度。在实战中，我们更多采用**"语义感知切分"**，配合一定的 Overlap 来保证上下文连贯性。

**选型逻辑**：重点聊了 Milvus 在大规模数据下的表现。不仅仅是看 Recall，更看重 **QPS 和稳定性**。特别是在混合检索场景下，如何平衡倒排索引和向量索引的权重，是落地的关键。

---

## 二、传统工程相关：高并发下的系统稳定性

### Q：高并发、分布式场景下的数据一致性问题？

**✅ 应对思路：**

Agent 本质上还是一个服务端应用。我结合之前的后端架构经验，聊了**延时双删、Binlog 异步订阅、服务间 ACK** 等方案。

这里我想表达的是：**Agent 的不稳定性已经够高了（模型幻觉），如果工程架构（数据库、缓存、工具调用）再出问题，那用户体验就是灾难级的。**

---

## 总结

面试核心考察点：
1. **RAG工程化**：不是调API就行，要懂分块策略、向量库选型、混合检索权重平衡
2. **传统工程能力**：Agent开发不只是写Prompt，高并发、数据一致性等后端基本功依然重要
3. **工程化落地能力**：大厂看重的是从Demo到生产的完整链路能力
