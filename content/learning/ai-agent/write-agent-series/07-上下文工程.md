---

之前的文章中我们聊到了 Agent 的系统提示词、工具调用、记忆系统。

其实这些归根到底，都是在给模型喂信息 — 也就是「上下文」。

Agent 能不能按照预期完成任务，除了模型本身的能力之外，最大的影响因素就是模型能看到什么信息。

新模型发布的时候，都会强调它能支持的上下文长度，比如最新的 Deepseek-V4：

上下文越长，意味着每次调用模型时能提供的信息就越多。

但不是喂给模型的信息越多越好。

首先，上下文越长，Token 消耗就越大，成本更高，响应也更慢。

其次，信息太多就会形成噪音，影响模型判断，反而导致任务失败。

所以我们需要在「给够信息」和「不让模型混乱」之间找到平衡 — 这就是上下文工程。

下面我们从简单到复杂，聊聊具体的上下文工程策略。

一、滑动截断

最简单的策略：当对话接近上下文上限时，只保留最近 N 轮，丢弃更早的内容。

粗暴但有效。

二、分层摘要

滑动截断有个问题 — 重要信息可能被丢掉了。

分层摘要的思路是：

• 最近 10 轮对话：完整保留原始内容，不压缩

• 第 11 到 40 轮：滚动详细摘要，提取关键信息（做了什么决策、调用了什么工具、得到什么结果、任务状态是什么）

• 40 轮之前：压缩为高层次摘要，保留高级目标、核心约束、主要中间结论

三、记忆系统

当滑动窗口和分层摘要依然无法满足长任务所需要的上下文信息时，我们需要把信息卸载到外部存储。

Agent 根据当前任务动态检索相关内容注入上下文。

这本质上是把上下文窗口从「一次性内存」变成了「可查询的知识库」。

核心实现

好了，聊完理论，我们来给 Agent 加上这些策略。

"""
