# 17 种 Agent 架构模式深度解析

> 来源：微信公众号《周末程序猿》linkxzhou | 基于 all-agentic-architectures + agno 框架
> **类型**：📚 参考资料（非面试题/面经）

## 核心观点

> Agent architecture 的本质不是 prompt engineering，也不是某个框架的 DSL，而是**控制流设计**。它应该能在任何体面的 agent框架里复现。

真正决定 agent 系统能不能落地的，通常不是模型回答是不是够好，而是：
- 状态有没有被正确建模
- 控制流有没有被显式表达
- 错误能不能被局部截断
- 副作用能不能被关进闸门
- 系统知不知道自己什么时候该停

---

## Agent 演化路径（9 次升级）

| 阶段 | 升级 |
|------|------|
| 1 | 单次生成 → 反思闭环 |
| 2 | 反思闭环 → 工具交互 |
| 3 | 工具交互 → 观察-行动循环 |
| 4 | 局部决策 → 显式规划 |
| 5 | 无验证执行 → 验证驱动重规划 |
| 6 | 单 agent → 多 agent 编排 |
| 7 | 短期上下文 → 长期记忆系统 |
| 8 | 线性推理 → 搜索、模拟与涌现计算 |
| 9 | 能做事 → 可信任 |

---

## 统一分析框架（6 个问题）

每种架构都用同一套问题拆解：

| # | 问题 | 说明 |
|---|------|------|
| 1 | 它要解决什么问题？ | 上一代架构哪里不够 |
| 2 | 它的 State 是什么？ | 新增了哪些字段，为什么必须存在 |
| 3 | 它的拓扑是什么？ | 线性链、循环、分叉汇聚、共享黑板、树搜索还是网格涌现 |
| 4 | 它的 Router 怎么工作？ | 固定边、条件边、动态调度、验证回路、人工审批 |
| 5 | 它的失败模式是什么？ | 架构最容易在哪个环节坏掉 |
| 6 | 什么时候该升级到下一种？ | 当前模式的能力边界在哪里 |

---

## 17 种架构模式详解

### 1. Reflection（反思闭环）

- **解决的问题**: 最小质量闭环，把生成过程拆成**生成**和**评估**两个不同职能的 pass
- **拓扑**: 线性双 pass（生成 → 评估 → 可能重生成）
- **Router**: 条件边（评估不通过则重生成）
- **失败模式**: 模型可能陷入无限反思循环
- **能力边界**: 解决质量，但不解决知识边界

### 2. Tool Use（工具使用）

- **解决的问题**: LLM 的知识边界问题，让模型能访问外部世界
- **拓扑**: 线性（生成 → 工具调用 → 结果消费）
- **Router**: 固定边
- **失败模式**: 工具调用格式错误、工具返回异常
- **能力边界**: 工具结果只被消费一次，不进入下一轮决策

### 3. ReAct（观察-行动循环）

- **解决的问题**: 让工具结果不只是被消费一次，而是进入下一轮决策
- **拓扑**: 循环（观察 → 思考 → 行动 → 观察...）
- **Router**: 循环边（直到 done）
- **失败模式**: 在线贪心策略，对复杂任务可能陷入局部最优
- **Agent 真正成形的地方**

### 4. Planning（规划）

- **解决的问题**: 需要顺序约束、步骤依赖和过程可追踪的任务
- **拓扑**: 先规划后执行（计划 → 按步骤执行）
- **Router**: 固定边（按规划执行）
- **失败模式**: 规划脱离实际、环境变化后规划失效
- **把控制流本身变成模型输出**

### 5. PEV（Plan-Execute-Verify，验证驱动重规划）

- **解决的问题**: API 会失败、搜索会返回噪音、数据会缺失、工具会超时
- **拓扑**: 规划 → 执行 → 验证 → (失败则重规划)
- **Router**: 验证回路（验证不通过则重规划）
- **失败模式**: 验证器本身可能误判
- **把"验证"提升为控制流的一等公民**

### 6. Multi-Agent Collaboration（多 Agent 协作）

- **解决的问题**: 认知分工，不同角色处理不同子任务
- **拓扑**: 分叉汇聚（主 agent 分配 → 多个专家 agent 并行 → 汇总）
- **Router**: 动态调度（主 agent 决定分配给哪个专家）
- **失败模式**: 通信开销、角色冲突、汇总逻辑复杂
- **核心不是"多个模型一起跑"，而是把认知分工、调度策略和冗余机制显式写进图结构**

### 7. Blackboard（共享黑板）

- **解决的问题**: 不要预先写死专家顺序，让专家根据黑板状态自主决定行动
- **拓扑**: 共享黑板（多个 agent 读写同一状态空间）
- **Router**: 条件边（基于黑板状态决定下一步）
- **失败模式**: 黑板状态冲突、死锁
- **共享状态成为系统中心**

### 8. Meta-Controller（元控制器）

- **解决的问题**: 请求路由分类（研究类、代码类、分析类等），一次性路由而非持续编排
- **拓扑**: 路由分发（入口分类 → 路由到专用 agent）
- **Router**: 动态调度（基于请求类型路由）
- **失败模式**: 路由分类错误
- **一次性路由，而不是持续编排**

### 9. Ensemble（集成）

- **解决的问题**: 同一个问题，单个 agent 不够可靠，需要冗余
- **拓扑**: 并行独立（多个 agent 独立解决同一问题 → 投票/聚合）
- **Router**: 投票/聚合
- **失败模式**: 多数投票被系统性偏差带偏
- **不是分工，而是冗余**

### 10. Episodic Memory（情景记忆）

- **解决的问题**: 记住过去的具体事件和经验
- **拓扑**: 状态外延（agent + 事件存储 + 检索）
- **Router**: 条件检索（基于当前状态检索相关历史）
- **失败模式**: 检索到不相关事件
- **记忆不是把对话塞回上下文**

### 11. Semantic Memory（语义记忆）

- **解决的问题**: 从经验中抽象出通用知识和规则
- **拓扑**: 状态外延（agent + 知识库 + 推理）
- **Router**: 条件检索
- **失败模式**: 知识过时、知识冲突
- **区分事件记忆和语义知识**

### 12. Graph / World-Model Memory（图谱/世界模型记忆）

- **解决的问题**: 向量检索能回答"哪段历史最像现在的问题"，但不能回答"这个实体和那个实体的关系是什么"
- **拓扑**: 图结构（实体 → 关系 → 实体）
- **Router**: 图遍历
- **失败模式**: 图构建错误、关系推理链断裂
- **当你需要的不是回忆，而是关系推理**

### 13. Tree-of-Thoughts（思维树搜索）

- **解决的问题**: 真正困难的问题不是"链不够长"，而是"路径会分叉，而且需要回溯"
- **拓扑**: 树搜索（多分支探索 → 评估 → 回溯/继续）
- **Router**: 搜索策略（BFS/DFS/启发式）
- **失败模式**: 搜索空间爆炸
- **不是让模型"想更多"，而是让系统"搜更多"**

### 14. Mental Loop / Simulator（心理模拟）

- **解决的问题**: 在机器人、交易、生产系统配置变更这类任务里，试错有代价
- **拓扑**: 内部模拟循环（在内部世界试错 → 评估 → 选择最优方案 → 外部执行）
- **Router**: 模拟评估
- **失败模式**: 模拟与现实差距大
- **行动之前，先在内部世界里试错**

### 15. Dry-Run Harness（试运行闸门）

- **解决的问题**: 把副作用关进闸门，防止 agent 造成不可逆损害
- **拓扑**: 试运行 → 审查 → 执行
- **Router**: 人工审批 / 安全策略检查
- **失败模式**: 试运行覆盖不全
- **真正把副作用关进闸门里**

### 16. Reflexive Metacognitive Agent（自反元认知）

- **解决的问题**: 前面的系统都在问"怎么把任务做完"，元认知问"我能不能做完"
- **拓扑**: 自我监控循环（任务执行 + 自我评估 + 策略调整）
- **Router**: 自我评估回路
- **失败模式**: 过度自信或过度保守
- **系统第一次显式思考自己的边界**

### 17. Self-Improvement Loop（自进化回路）

- **解决的问题**: Reflection 只有一次 critique pass，高质量输出需要：生成 → 评估 → 学习 → 再生成
- **拓扑**: 长期进化循环（多次迭代 + 知识持久化）
- **Router**: 多轮迭代
- **失败模式**: 退化循环（越改越差）
- **进化回路**

### 18. Cellular Automata（细胞自动机）

- **解决的问题**: 前面所有架构都默认存在中心 agent，细胞自动机让智能从局部规则中涌现
- **拓扑**: 网格涌现（无中心，每个 cell 按局部规则行动）
- **Router**: 局部规则
- **失败模式**: 全局行为不可预测
- **LLM 退出主循环，智能从局部规则中涌现**

## Evaluator 不是可选项

至少有五类 evaluator：
1. **正确性评估**: 输出是否正确
2. **安全性评估**: 输出是否安全
3. **效率评估**: 是否用了最少步骤
4. **一致性评估**: 多步之间是否自洽
5. **用户满意度**: 是否满足用户真实需求

## 对面试的启示

1. **Agent 架构是面试热点**: 17 种模式覆盖了从简单到复杂的所有场景
2. **理解"为什么"比"是什么"重要**: 每次架构升级都是为了解决上一代的不足
3. **控制流设计是核心**: 不是 prompt engineering，而是状态建模 + 控制流 + 错误处理
4. **agno 框架值得了解**: 比 langchain/langgraph 更简洁，用 Agent + Workflow + Step + Loop + Router 五个抽象就能表达所有架构
5. **系统设计能力**: 面试官会问"你的 agent 系统用哪种架构？为什么？失败模式是什么？"
