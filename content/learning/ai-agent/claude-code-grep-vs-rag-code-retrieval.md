# 为什么 Claude Code 用 grep 不用 RAG 检索代码？

**来源**: 微信公众号  
**日期**: 2026-05-25  
**原文标题**: 京东面试官："为什么Claude Code不用Rag索引代码，而是用grep"，我笑了："RAG的Chunking放到代码里，最怕把结构切断"，面试官点头

---

## 核心结论

> **这不是 grep 赢了 RAG，而是场景选型问题。**
> 
> 代码检索更需要**实时性、精确性、可解释性和多轮决策**。RAG 不适合作为代码检索的主路径，但在跨仓语义发现和知识沉淀场景仍有价值。

---

## 一、代码检索 vs 文档检索 — 根本不是一类问题

### 文档检索适合 RAG
- 文档相对稳定
- 用户问题偏语义匹配
- 答案可以由多个片段综合得出

### 代码检索特点
| 检索目标 | 示例 | 要求 |
|---------|------|------|
| **找符号（Symbol Lookup）** | `getUserById`、`UserService` | 精确匹配，不需要语义相似 |
| **找入口** | "登录逻辑在哪" | 关键词探索，不是一次向量召回 |
| **找关系（调用链）** | 字段从接口到入库的路径 | controller → service → dao 一路读 |
| **找变化** | "改了支付状态为什么测试挂了" | 最新文件内容、git diff |

**代码检索的核心**不是"找相似内容"，而是：
> 在一个不断变化的结构化工程里，**快速定位当前任务真正需要的最小上下文**。

## 二、传统 RAG 检索代码的 4 个硬伤

### 1. Chunking 破坏代码结构
```
按固定行数切 → 函数被切断
按函数切     → 类上下文丢失
按类切       → 文件太大
按 AST 切    → 多语言工程复杂度高
```
函数的意义依赖于：参数类型、返回类型、调用方、被调用函数、配置项、DB schema、测试用例。Chunk 割裂上下文 → 模型基于残缺上下文改代码 → **幻觉概率上升**。

### 2. 向量相似 ≠ 代码正确
```java
// 你要改 validateToken()
// 向量库可能召回：
verifyToken()    // 语义很像，但不是目标
decodeToken()    // 语义很像，但不是目标
refreshToken()   // 语义很像，但不是目标
validateSession()// 语义很像，但不是目标
```
**语义相似是优势，但精确匹配是底线。** 没有底线，召回越多越乱。

### 3. 索引滞后伤害实时开发
知识库 RAG 可以每天更新一次。代码不行。Claude Code 刚改完文件，下一步就要读它验证。RAG 存在尴尬问题：
- 索引更新了吗？没更新 → 查到旧代码
- 谁来决定哪些 chunk 失效？跨文件引用怎么同步？
- 刚改了函数签名，调用方 chunk 要不要重算？

**grep 和 Read 是现读磁盘，没有缓存一致性问题。**

### 4. Top-K 是一次性下注，Agent 需要边走边看
```
RAG:       用户问题 → 向量召回 Top-K → 模型回答（一次召回，错了全歪）
Agent:     搜关键词 → 没搜到换词 → 太多缩小目录 → 
           看到文件发现调用另一个函数 → 跟过去 → 看到测试报错再回头查
```
**RAG 像发试卷，Agent 检索像现场排查。**

## 三、Claude Code 为什么拆成 Glob、Grep、Read 三件套

### 为什么不直接给万能 Bash？
1. **权限边界清楚** — Grep/Read 是只读，Write/Edit 是写操作，安全可控
2. **返回结果可控** — 文件路径、行号、匹配内容、截断策略结构化，不是纯文本
3. **工具描述进 Prompt** — 教模型什么时候用、参数怎么填、什么情况不要用

### 三件套分工

| 工具 | 能力 | 解决什么问题 |
|------|------|-------------|
| **Glob** | 按文件名模式找文件 | **文件入口** — 找 `**/*auth*`、`**/*.route.ts` |
| **Grep** | 按内容搜索（精确+正则） | **符号定位** — 搜 `getUserById`、`rg "POST /login"` |
| **Read** | 按需读取文件内容 | **上下文边界** — 只读命中行附近几十行，不整文件倒 |

### 标准探索链路示例
> "登录接口加一个验证码校验"

```
Glob → 找路由和登录相关文件   (**/*auth*, **/*login*)
Grep → 搜 login, signin, token
Read → 读取命中文件附近逻辑
发现登录调用了 AuthService
Grep → AuthService
Read → service 实现
找到测试文件 → 修改代码 → 跑测试验证
```

这不是一次检索，是**不断收敛的链路**。每一步都在问："我现在知道了什么？下一步最该看哪里？"

## 四、真正关键的是 Agent Loop，不是某个工具

```python
# 伪代码：Agent 多轮检索
while not goal_achieved:
    action = model.decide_next_action()   # 模型思考下一步
    result = call_tool(action)             # 调用工具
    model.observe(result)                  # 模型观察结果
    # 模型判断：继续搜？缩小范围？读文件？改代码？
```

### RAG vs Agent 检索
| 维度 | 传统 RAG | Agent 多轮检索 |
|------|---------|--------------|
| 召回时机 | 一次性 Top-K | 动态搜索，边看边改方向 |
| 错误恢复 | 召回错了后面全歪 | 看结果判断 → 换关键词/缩小目录/跟调用链 |
| 迭代性 | 无 | 工具结果不是答案，而是下一轮决策的证据 |

## 五、子 Agent：把探索过程隔离出去

### 为什么需要子 Agent？
> "如果主 Agent 一直 Grep、Read，上下文不会爆吗？"

会。所以 Claude Code 的 **Task** 工具让子 Agent 去处理复杂探索：
- **子 Agent 有独立上下文窗口** — 搜索结果不会污染主对话
- **权限隔离** — 探索型子 Agent 只给只读工具（Glob/Grep/Read）
- **压缩结论返回** — 子 Agent 把自己的调研结论压缩后返回给主 Agent

### 上下文污染的例子
```java
// 主 Agent 要改一个鉴权逻辑
// 如果自己探索：搜 auth → 搜 login → 搜 jwt → 搜 middleware →
// 读路由 → 读 service → 读配置 → 读测试 → 读拦截器
// 大量中间结果塞满上下文，真正写代码时注意力被冲散

// 更好的方式：
// 主 Agent 派子 Agent → "调研登录链路，返回关键文件和修改点"
// 子 Agent 在自己上下文里随便搜 → 返回压缩结论
// 主 Agent 基于结论动手改
```

## 六、grep 不是银弹 — RAG 什么时候仍然有价值

### RAG/混合检索仍然适合的场景

| 场景 | 说明 | 方案 |
|------|------|------|
| **超大规模跨仓库检索** | 几十个仓库、几亿行代码 | 关键词倒排索引 + 符号索引 + AST 索引 + 向量索引 = **混合检索** |
| **模糊需求探索** | "找一下和权限收敛相关的逻辑" | **向量做发现，grep 做确认** |
| **代码+文档综合问答** | "为什么订单状态分成 PAID 和 CONFIRMED？" | RAG 检索设计文档、ADR、Wiki |
| **长期记忆/团队知识沉淀** | 常见故障原因、历史架构决策、工程实践经验 | RAG 承载长期知识 |

### 最终选型结论
> **代码实时修改主路径 → grep/Glob/Read**
> **跨仓语义发现和知识背景 → RAG 或混合检索**

## 七、面试回答模板（三层结构）

### 第一层：代码检索不是文档检索
> "代码检索和知识库文档检索不一样。文档 RAG 关注语义相关，但代码检索经常是符号定位、文件入口、调用链追踪和最新内容读取。精确匹配比语义相似更重要。"

### 第二层：传统 RAG 在代码场景的结构性问题
> "第一，chunking 破坏函数/类/调用链结构。第二，向量召回是近似匹配，代码要准确目标。第三，代码每天在变，索引容易滞后。第四，Top-K 是一次性召回，写代码需要不断调整方向。"

### 第三层（升维）：Claude Code 的关键不是 grep，而是 Agent Loop
> "Claude Code 把 Grep、Glob、Read 放进 Agent Loop。模型先搜，再读，再根据结果决定下一步。这种多轮检索更像程序员现场排查问题。复杂探索交给子 Agent 在独立上下文里搜索和总结，避免污染主 Agent 上下文。所以这不是 grep 和 RAG 谁高级，而是主路径选型。"

### 高频追问准备

**追问1：grep 搜不到语义怎么办？**
> "多轮关键词改写 + 文件结构探索 + 调用链阅读来弥补。跨仓库模糊语义发现时引入混合检索：**向量做发现，grep 做确认**。"

**追问2：为什么不用 AST 索引？**
> "AST 索引对符号跳转、引用查找很有价值，但工程成本更高、多语言支持复杂、实时更新麻烦。Claude Code 的核心场景是快速适配任意项目，通用文件模式搜索更轻量。AST 索引可以作为增强能力。"

**追问3：RAG 能不能做增量更新？**
> "能做，但增量更新本身很复杂——代码改动影响 chunk、符号引用、调用链。实时编码场景现读磁盘天然避免一致性问题。公司级知识库再做增量索引是合理的。"

**追问4：一句话总结 Claude Code 设计哲学？**
> "**把确定性工具交给模型，把检索决策权还给模型，让模型在多轮工具反馈中逐步逼近答案。**"

---

## 八、核心启发

- **AI 编程工具的核心**不是把整个仓库喂给模型，而是让模型像靠谱工程师一样：先找入口，再看证据，再跟调用链，再动手改
- **别迷信复杂架构**，能用简单工具解决的问题，先把简单工具用到极致
- **代码检索最怕的不是"搜不到很多相关内容"，而是搜到一堆看起来相关、实际上不能改的内容**
