我写第 1 篇的时候聊了聊 AgentScope 的定位：一个面向 JVM 的 Agent 框架。但框架终归是框架，最终你打交道最多的还是 Agent。所以第 2 篇我们直接看 Agent 体系。

AgentScope 的 Agent 分两层。底层是 

ReActAgent

，一个纯的推理-行动循环引擎；上层是 

HarnessAgent

，它在 ReActAgent 外面叠了一层工程化基础设施。写这篇文章的时候我把这两部分的源码都翻了一遍，来一个一个说。

Agent 接口：最小的约定

先看最顶层。框架定义了一个 

Agent

 接口，聚合了三个子接口的能力：

CallableAgent

 — 接收消息，返回 

Mono<Msg>

StreamableAgent

 — 支持流式事件

ObservableAgent

 — 只接收不回复（多 Agent 协作用）

接口在 

agent-core/src/main/java/io/agentscope/core/agent/Agent.java

。注意一个设计决定：内存管理不在 

Agent

 接口里。框架的原则是"需要内存的 Agent 自己实现"，所以 

Agent

 只关心收发消息和生命周期，具体的记忆逻辑下沉到 

ReActAgent

。

AgentBase

 是抽象基类，在 

agent-core/src/main/java/io/agentscope/core/agent/AgentBase.java

。它提供了 hook 集成、消息订阅、中断处理、调用序列化这些公共能力。

ReActAgent

 继承 

AgentBase

。

ReActAgent：推理-行动循环

ReActAgent

 在 

agent-core/src/main/java/io/agentscope/core/ReActAgent.java

，大约 4500 行。它的核心思想一句话就能说清楚：

循环调用 LLM，每次推理出一个行动，执行行动后把观察结果喂回下一轮推理，直到任务完成或达到迭代上限

。

acting

 阶段拿到上一步生成的 tool call，执行，然后把结果写回 context，再递归调用

 启动下一轮。如果工具调用挂起了（pending，比如等待人工确认），就返回

 暂停循环。

为什么叫 ReAct

“ReAct” 这个词来自 Google 那篇论文 

ReAct: Synergizing Reasoning and Acting in Language Models

。它和普通的 CoT（Chain of Thought）是两回事。CoT 只在推理空间里打转，ReAct 会在推理和行动之间来回切换。AgentScope 的实现忠实于这个模式：每轮迭代先想一想，再做一做，再看结果，再想下一步。

现在看 HarnessAgent。它在

生命周期钩子

2.0 引入了 Middleware 体系，替换了 1.x 的 Hook。五个 stage：

onAgent

 — 整个 call 的入口

onReasoning

 — 每次 LLM 调用前

onActing

 — 每次工具执行前

onModelCall

 — 实际网络请求前后

onSystemPrompt

 — 系统提示词组装

Middleware 用 

MiddlewareChain.build()

 串联：

// 以 onReasoning 为例

Function<ReasoningInput, Flux<AgentEvent>> reasoningCore = ri -> reasoningStream(...);

Flux<AgentEvent> stream = MiddlewareChain.build(

    middlewares, ReActAgent.

this

, rc,

    MiddlewareBase::onReasoning,

    reasoningCore

).apply(

new

 ReasoningInput

(modelInput, tools, options));

每个 middleware 可以选择跳过、修改入参、或者提前截停。这种设计比继承重写要灵活得多——想加能力就加一个 middleware，不用动核心类。

本质区别

一句话总结：

ReActAgent 是推理引擎，HarnessAgent 是运行时平台

。

你写一个聊天机器人、一个单轮工具调用，用 ReActAgent 完全够。但如果你需要：

agent 的记忆跨进程恢复

context 太长时自动压缩

一个 workspace 管理 agent 的身份和知识

子代理拆分复杂任务

沙箱环境执行代码

那就上 HarnessAgent。它的所有能力都挂载在 middleware 上，不动 ReActAgent 一行代码。

还有一点值得一提：HarnessAgent 的 builder 同时暴露了 

inner

（即 ReActAgent.Builder），所以你把 ReActAgent 的配置平滑迁移过来几乎不用改代码——

HarnessAgent.builder().name(...).model(...)

 跟 

ReActAgent.builder().name(...).model(...)

 写法一模一样，完全正交。

 这些 lifecycle hook 怎么写自己的中间件。
