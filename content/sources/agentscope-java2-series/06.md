今天我们探讨一下在整个 Agent 运行时里最让我觉得"设计得好"的一个组件——

Middleware 中间件

。

为什么需要中间件？

先想一个问题：你写了一个 ReActAgent，跑得挺好。突然有一天，产品经理说"每个模型调用都要记录耗时"，第二天又说"把用户的请求 ID 透传到每个工具里"，第三天"给 system prompt 动态加点上下文"。

没有中间件，你就得改 ReActAgent 的源码，或者在每个业务场景里自己包一层。一次两次还行，长了就变成每个项目一套 fork。

中间件要解决的就是这个——

在不改 Agent 核心代码的前提下，让你能在执行流程的关键位置插入自定义逻辑

。做过 Web 开发的朋友看到这个应该很熟悉，这本质上就是 Web 框架里 Filter/Interceptor 那套东西，只不过 Agent 的执行流程有自己的生命周期。

两种拦截模式

翻阅 AgentScope Java 2.0 的源码，发现它设计了两种模式：

Onion（洋葱模式）

mw1_pre → mw2_pre → inner → mw2_post → mw1_post

外层包裹内层，每层可以在调用 

next.apply(input)

 之前和之后分别插逻辑。流式场景下，每个事件也是先经过内层处理再往外层冒泡。

Transformer（管道模式）

和洋葱不同，这里没有"内层外层"的概念。多个中间件串成一个管道，前一个的输出直接喂给后一个。目前这个模式只用在 

onSystemPrompt

 上。

五个钩子位置

Agent 的执行流程被切成了 5 个可以下钩子的点，稍微调整了下官网的图，这样看着更清楚些：

onAgent

  └── ReAct

 loop

 (每轮)

/

       ├── onReasoning

       │    ├── onSystemPrompt (组装 system prompt)

       │    └── onModelCall (模型 API 调用)

       └── onActing (执行工具)

钩子

类型

拦截什么

onAgent

Onion

整个 reply 流程，包含所有 ReAct 轮次

onReasoning

Onion

单轮推理（拼消息 → 调模型 → 解析输出）

onActing

Onion

执行工具调用

onModelCall

Onion

最底层的 ChatModel API 调用

onSystemPrompt

Transformer

组装系统提示词时

每个钩子都有自己的 Input Record，携带当前上下文的数据。后面源码部分会细看。

所有方法都有 default 实现（直接透传 

next

），所以你只需要 override 关心的方法就行。每个方法都接收 

Agent

、

RuntimeContext

、当前输入的 Record、以及 

next

 函数。

设计上比较好的几点：

每个钩子的输入是独立的 Record 类型

，不会搞混参数。比如 

onActing

 拿到的是 

ActingInput(List<ToolUseBlock>)

，而 

onModelCall

 拿到的是 

ModelCallInput(List<Msg>, List<ToolSchema>, GenerateOptions, Model)

。

返回值是 

Flux<AgentEvent>

，天然支持流式——你可以在返回的 Flux 上做 

doOnNext

、

flatMap

、

map

 来观察或改写事件流。

RuntimeContext

 被传进了每个方法，这我在 Day 2 提到过，它是 per-call 的上下文，线程安全，中间件和工具之间可以通过它共享数据。

MiddlewareChain —— 链的组装逻辑

文件路径：

agentscope-core/src/main/java/io/agentscope/core/middleware/MiddlewareChain.java

public

 static

 <I> Function<I, Flux<AgentEvent>>

 build

(

        List<? extends MiddlewareBase> middlewares,

        Agent agent,

        RuntimeContext ctx,

        MiddlewareMethod<I> method,

        Function<I, Flux<AgentEvent>> core)

 {

    if

 (middlewares ==

 null

 || middlewares.isEmpty()) {

        return

 core;

    }

    Function<I, Flux<AgentEvent>> chain = core;

    for

 (

int

 i

 =

 middlewares.size() -

 1

; i >=

 0

; i--) {

        MiddlewareBase

 mw

 =

 middlewares.get(i);

        Function<I, Flux<AgentEvent>> next = chain;

        chain = input -> method.apply(mw, agent, ctx, input, next);

    }

    return

 chain;

}

这段代码我看了两遍才完全理解它的巧妙。它从列表

末尾

开始往前构建——先让最后一个中间件包裹 core，然后逐个往前包。这样第一个注册的中间件就是最外层，符合直觉。

比如你注册了 

[mw1, mw2]

：

先 

mw2

 包裹 

core

再 

mw1

 包裹 

(mw2 → core)

最后返回的 

chain

 函数调用时就是 

mw1_pre → mw2_pre → core → mw2_post → mw1_post

。

文件路径：

agentscope-core/src/main/java/io/agentscope/core/middleware/TaskReminderMiddleware.java

这个中间件和任务列表工具 

TodoTools

 配合使用。它干了两个事：

onSystemPrompt

onReasoning

：每次推理前，把当前的待办列表渲染成

@Override

public

 Flux<AgentEvent>

 onReasoning

(

        Agent agent, RuntimeContext ctx, ReasoningInput input,

        Function<ReasoningInput, Flux<AgentEvent>> next)

 {

    AgentState

 state

 =

 RuntimeContext.resolveAgentState(ctx, agent);

    List<Task> tasks = state ==

 null

 ? List.of() : state.getTasksContext().getTasks();

    if

 (tasks.isEmpty()) {

        return

 next.apply(input);

    }

    // 构造 reminder 消息

    Msg

 reminder

 =

 Msg.builder()

            .role(MsgRole.USER)

            .name(

"system"

)

            .content(TextBlock.builder().text(render(tasks)).build())

            .metadata(Map.of(Msg.METADATA_SYNTHETIC,

 true

, Msg.METADATA_REMINDER_KIND,

 "todo_state"

))

            .build();

    // 追加到消息列表

    List<Msg> messages =

 new

 ArrayList

<>(input.messages());

    messages.add(reminder);

    return

 next.apply(

new

 ReasoningInput

(messages, input.tools(), input.options()));

}

代码注释里说得很直白：这个 reminder 是

瞬态

的，不会写进 

AgentState.context

，所以不会被持久化或压缩。每次推理前重新注入，保证模型始终看到最新的列表——即使之前的工具调用结果已经被压缩掉了。

文件路径：

agentscope-core/src/main/java/io/agentscope/core/tracing/OtelTracingMiddleware.java

这个中间件通过 OpenTelemetry 为标准 GenAI 属性打 span。三个钩子各产生一种 span：

public

(

    return

 =

 resolveOtelContext(ctxView);

        Span

 span

 =

 getTracer().spanBuilder(

"chat "

 + modelName)

                .setParent(parentContext)

                .setAttribute(

"gen_ai.operation.name"

,

 "chat"

)

                .setAttribute(

"gen_ai.request.model"

, modelName)

                .startSpan();

        // ...

        return

                .doOnNext(event -> {

这里用

 来处理 Reactor 的线程切换问题——span 可能在某个线程创建，但真正的模型调用在另一个线程执行。通过

 在 Reactor pipeline 里传播 OTel Context，保证父子 span 关系不乱。

注册机制和执行顺序

Middleware 通过 Builder 注册，支持两种方式：

// 追加一个

.middleware(

new

 LoggingMiddleware

())

// 传入列表

.middlewares(List.of(

new

 AuditMiddleware

(),

 new

 TraceMiddleware

()))

middleware(...)

（单数）追加一个；

middlewares(...)

（复数）接受一个 List。

执行顺序：

Onion 钩子

：列表中第一个是最外层。

[mw1, mw2]

 → 

mw1_pre → mw2_pre → inner → mw2_post → mw1_post

Transformer 钩子

：从左到右串联。

[mw1, mw2]

 → 

originalPrompt → mw1 → mw2 → final

如果你注册了多个中间件，每个钩子的洋葱链是

分别构建

的。也就是说 

onAgent

 的链只包含实现了 

onAgent

 的中间件；未实现的方法直接跳过，零开销。

和 HarnessAgent 的关系

HarnessAgent

 是 

ReActAgent

 的增强版，在 Builder 上同样有 

.middlewares(...)

 方法。也就是说你在 

ReActAgent

 上写的所有中间件，切到 

HarnessAgent

 后不用改一行代码。

Harness 本身的 Workspace、Compaction、Skill 这些能力，大多也是通过内置中间件实现的——只是这些中间件是框架内部注册的，对外不可见。

总结

AgentScope Java 2.0 的中间件设计，是我在这个框架里最喜欢的一部分。说几个我觉得做得好的地方：

分层清晰

。5 个钩子位置对应 Agent 执行的完整生命周期，不多不少。

接口干净

。每个钩子都有自己的 Input Record，方法签名统一，默认实现直接透传。

零成本抽象

。没实现的钩子跳过，没有反射或代理开销。

流式友好

。返回 

Flux<AgentEvent>

，Observability 相关中间件可以直接在 stream 上做 doOnNext。

RuntimeContext 贯穿始终

。中间件之间、中间件和工具之间可以通过它共享数据，不用搞全局变量。

接下来我们继续探讨 Context & AgentState，看看 AgentScope 2.0 里状态管理到底是怎么设计的。下篇见。
