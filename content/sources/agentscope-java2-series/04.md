Agent 的核心是 LLM，但无论 Agent 多聪明，底层总得有个模型在干活。

这篇就把 AgentScope Java 2.0 的模型层彻底讲清楚。翻了文档也翻了源码，争取一次性把下面的问题聊透：

框架怎么统一管理那么多家 LLM 的 API？

dashscope:qwen-max

 这种字符串是怎么变成真实模型调用的？

如果你公司有自己的模型推理服务，怎么接进来？

两层架构：Credential 和 ChatModel

先说设计思路。

AgentScope 把模型层拆成了两层：上面是

Credential

（凭证），下面是

ChatModel

（模型实例）。

CredentialBase/

└── ChatModelBase/

    ├── DashScopeChatModel

    ├── OpenAIChatModel

    ├── AnthropicChatModel

    ├── GeminiChatModel

    └── OllamaChatModel

Credential 管的是"怎么连"——API Key、baseUrl、endpointType 这些连接参数。ChatModel 管的是"怎么调"——发什么请求、收什么响应、流式怎么处理。

我一开始觉得这个分层有点多余，但后来理解了：前端场景里，用户先配一个凭证（比如输 DashScope API Key），然后系统自动拉出该凭证下可用的模型列表让用户选。Credential 提供的

listModels()

 方法就是干这个的。两层分离后，前端只需要处理一次认证，后续模型选择就变成一个下拉框的事。

关于 Formatter

每个模型厂商的消息格式不一样——OpenAI 用

AgentScope 用

厂商

单 Agent 模式下，每个

 接口，通过

formatter(...)

 builder 方法注入。

总结

AgentScope 2.0 的模型层合起来就是：

Credential

 管连接参数，

ChatModel

 管推理调用，两层分离

ModelRegistry

 做字符串到实例的映射，支持命名注册、用户工厂、SPI 三种扩展方式

SPI 机制

让每个厂商独立发布 JAR，用 Java 标准的

ServiceLoader

 做自动发现

Formatter

 屏蔽各家 API 格式差异

国产模型 DashScope 原生支持，DeepSeek/智谱/Kimi 走 OpenAI 兼容模式

自建推理服务通过 SPI 三步就能接入，跟内置模型一样的体验
