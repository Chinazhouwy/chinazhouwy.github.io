Agent 之间的语言与通信协议

前两篇我们搭建了环境，同时也认识了 Agent 体系，但无论 Agent 多聪明，它们之间怎么交流？用户发给 Agent 的消息、Agent 返回的回复、工具调用的参数和结果——这些数据到底长什么样？

通过阅读源码发现，AgentScope 的消息模型比我想象得要讲究。它没有简单地把消息当成一个字符串，而是设计了一套完整的类型系统。今天我们就来拆开看看。

消息模型：Msg 和它的四个分身

先说结论：AgentScope 里所有对话数据都用

Msg

 这个类表达。不管是你问的问题、Agent 的思考过程、还是工具返回的结果，最终都被包装成一个

Msg

。

Msg

 的核心字段就这几个：

说明

String

全局唯一 ID

name

String

发送者名字

role

USER/ASSISTANT/SYSTEM/TOOL

content

List<ContentBlock>

消息内容，用 ContentBlock 列表表达

metadata

Map<String, Object>

元数据

timestamp

String

yyyy-MM-dd HH:mm:ss.SSS 格式

有意思的是

content

 字段——它不是字符串，而是一个

ContentBlock

 列表。这就意味着一条消息可以同时包含文本、图片、工具调用等不同类型的内容。

每个子类都固定了自己的

，并且对内容做了限制：

UserMessage

：USER 角色，只能包含文本、DataBlock、ImageBlock、AudioBlock、VideoBlock

：ASSISTANT 角色，什么 ContentBlock 都能装

SystemMessage

：SYSTEM 角色，只能装 TextBlock

ToolResultMessage

：TOOL 角色，专门装 ToolResultBlock

USER 消息就是你发的问题和附件——文本、图片、音频、视频。ASSISTANT 是 Agent 的回复，所以什么都能装，包括思考过程和工具调用。SYSTEM 只能是纯文本指令。

ContentBlock：Java sealed class 的妙用

ContentBlock 是 AgentScope 2.0 消息模型中最值得关注的设计。它是一个

sealed class

（Java 17+ 特性），限制了所有可能的子类型：

// 文件：agentscope-core/src/main/java/io/agentscope/core/message/ContentBlock.java

@JsonTypeInfo(use = ... , property = "type")

@JsonSubTypes({

    @JsonSubTypes.Type(value = TextBlock.class, name = "text"),

    @JsonSubTypes.Type(value = ThinkingBlock.class, name = "thinking"),

    @JsonSubTypes.Type(value = ImageBlock.class, name = "image"),

    @JsonSubTypes.Type(value = AudioBlock.class, name = "audio"),

    @JsonSubTypes.Type(value = VideoBlock.class, name = "video"),

    @JsonSubTypes.Type(value = ToolUseBlock.class, name = "tool_use"),

    @JsonSubTypes.Type(value = ToolResultBlock.class, name = "tool_result"),

    @JsonSubTypes.Type(value = HintBlock.class, name = "hint"),

    @JsonSubTypes.Type(value = DataBlock.class, name = "data")

})

public

 sealed

 class

 ContentBlock

 implements

 State

    permits

 TextBlock, ImageBlock, AudioBlock, VideoBlock,

            ThinkingBlock, ToolUseBlock, ToolResultBlock, HintBlock, DataBlock {}

用

sealed class

 而不是普通抽象类的好处——编译器知道所有子类，

switch

 模式匹配时可以穷举检查，少了一个 else 分支编译就报错，不会漏掉处理。

九种 ContentBlock 一览

Block 类型

用途

可用角色

TextBlock

纯文本对话、指令、描述文字

USER, ASSISTANT, SYSTEM

DataBlock

通用二进制多媒体块，通过 mediaType 区分图 / 音 / 视频，新项目推荐使用

USER, ASSISTANT

ImageBlock

历史遗留兼容图片类型，仅用于兼容存量数据，新业务禁止使用

USER

AudioBlock

历史遗留兼容音频类型，仅用于兼容存量数据，新业务禁止使用

USER

VideoBlock

历史遗留兼容视频类型，仅用于兼容存量数据，新业务禁止使用

USER

ThinkingBlock

模型推理过程、思维链、内部思考日志，仅模型输出

ASSISTANT

ToolUseBlock

模型发起工具调用请求、参数信息

ASSISTANT

ToolResultBlock

工具执行返回结果、工具输出数据

ASSISTANT

HintBlock

Agent 侧注入的临时上下文辅助提示，用于辅助模型推理

ASSISTANT

注意 DataBlock 是 v2 引入的"统一媒体块"，把原来分散的 ImageBlock/AudioBlock/VideoBlock 统一了。新代码应该用 DataBlock，老代码还能继续跑。

Event 事件流：实时 U I 的基石

Msg 是"完成态"的数据——Agent 处理完一轮交互后产出的完整消息。但如果要在界面上实时展示 Agent 的推理过程（就像 ChatGPT 那样一个字一个字往外蹦），就需要事件流。

事件体系的核心是

AgentEvent

 抽象类，同样用 Jackson 多态定义了大量子类型：

// 文件：agentscope-core/src/main/java/io/agentscope/core/event/AgentEvent.java

@JsonSubTypes({

    @JsonSubTypes.Type(value = AgentStartEvent.class, name = "AGENT_START"),

    @JsonSubTypes.Type(value = AgentEndEvent.class, name = "AGENT_END"),

    @JsonSubTypes.Type(value = TextBlockStartEvent.class, name = "TEXT_BLOCK_START"),

    @JsonSubTypes.Type(value = TextBlockDeltaEvent.class, name = "TEXT_BLOCK_DELTA"),

    @JsonSubTypes.Type(value = TextBlockEndEvent.class, name = "TEXT_BLOCK_END"),

    // ... 还有 THINKING_BLOCK_*、DATA_BLOCK_*、TOOL_CALL_* 等 20+ 种事件

})

public

 abstract

 class

 AgentEvent

 {

    private

 final

 String id;

    private

 final

 String createdAt;

    private

 String source;

  // 子 Agent 事件用 "main/researcher" 表示来源路径

}

事件遵循

start → delta(×N) → end

 三段式模式。比如文本输出：

TextBlockStartEvent

 — 一个新文本块开始，带

replyId

 和

blockId

TextBlockDeltaEvent

 — 增量文本，每次返回一部分（

getDelta()

）

TextBlockEndEvent

 — 文本块结束

一个典型的 Agent 调用产生的事件序列就像这样：

AgentStartEvent

  ├─ ModelCallStartEvent

  │   ├─ ThinkingBlockStartEvent →

 ThinkingBlockDeltaEvent

(×N) → ThinkingBlockEndEvent

  │   ├─ TextBlockStartEvent →

 TextBlockDeltaEvent

(×N) → TextBlockEndEvent

  │   └─ ToolCallStartEvent →

 ToolCallDeltaEvent

(×N) → ToolCallEndEvent

  ├─ ModelCallEndEvent

  ├─ ToolResultStartEvent →

 ToolResultTextDeltaEvent

(×N) → ToolResultEndEvent

  └─ AgentEndEvent

每个事件的

replyId

 把它们关联到同一个回复消息上。最终这些事件会"坍缩"成一个完整的 AssistantMessage。

我的一点体会

读这段源码给我最大的感受是设计上的克制。消息模型没有搞出一大堆花哨的接口和工厂，就是用四个角色子类 + Builder + ContentBlock 搞定。

sealed class

 用得恰到好处，既限定了 ContentBlock 的种类，又保持了扩展的可能。

Event 体系的设计也挺有意思——它和 Message 是"流"与"态"的关系。Agent 运行过程中吐出事件流（Stream），最终汇聚成一个完整的消息（State）。一个管实时展示，一个管持久化和上下文传递，分工明确。
