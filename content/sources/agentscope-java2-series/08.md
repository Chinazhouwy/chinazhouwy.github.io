Agent 能调工具是好事，但如果不加以限制，调什么工具都行，那就是灾难了。想想看，如果你写了个 Agent 帮你在服务器上做运维，它突然起意执行了 

rm -rf /

——大结局。

权限系统就是干这个的：拦住不该干的，放行该干的，拿不准的就问人。

三层决策模型

AgentScope 的权限系统不在 Agent 外面套一层壳，而是嵌在每次工具调用的路径上。每次 Agent 决定调一个工具，先过权限引擎，引擎给三个答案之一：

决策

含义

ALLOW

放行，直接执行

DENY

拒绝，工具调用被吃掉

ASK

暂停，等人拍板

引擎的决策逻辑有三层，按优先级从高到低：

Rules（规则）

 — 人工配置的显式规则，分 ALLOW / DENY / ASK 三种，DENY 优先级最高

Built-in Checks（内置检查）

 — 工具自己实现对传入参数的安全判断，连 BYPASS 模式都绕不过

Mode（模式）

 — 全局兜底策略，没规则匹配时按模式处理

看官网的决策流程图最清楚：

工具调用 → Deny规则? → 是 → DENY

         → Ask规则?  → 是 → ASK（等用户确认）

         → 工具自检 → 拒绝/读保护

         → Allow规则? → 是 → ALLOW

         → 模式兜底 → BYPASS → ALLOW

                    → DONT_ASK → DENY

                    → DEFAULT → ASK

官网文档地址：Permission System

文件

这是最关键的 335 行。它的

public

 Mono<PermissionDecision>

 checkPermission

(ToolBase tool, Map<String, Object> toolInput)

 {

    // 1. Deny 规则（最高优先级）

    PermissionDecision

 denyDecision

 =

 checkDenyRules(tool, input);

    if

 (denyDecision !=

 null

)

 return

 Mono.just(denyDecision);

    // 2. Ask 规则

    PermissionDecision

 askDecision

 =

 checkAskRules(tool, input);

    if

 (askDecision !=

 null

)

 return

 Mono.just(askDecision.withSuggestedRules(...));

    // 3. 工具自检（BYPASS 免疫）

    return

 toolCheckPermissions(tool, input)

            .flatMap(toolDecision -> {

                if

 (toolDecision.getBehavior() == DENY)

 return

 Mono.just(toolDecision);

                if

 (toolDecision.getBehavior() == ALLOW)

 return

 Mono.just(toolDecision);

                if

 (toolDecision.getBehavior() == ASK &&

 "safety"

.contains(...))

 return

 ...;

                return

 continueAfterToolCheck(tool, input);

            })

            .switchIfEmpty(continueAfterToolCheck(tool, input));

}

注意第 3 步的"BYPASS 免疫"——工具自检返回的 DENY 和 ASK（涉及安全）即使 BYPASS 模式也拦得住。这就是 

rm -rf /

 最后一个防线的实现。

continueAfterToolCheck

 处理 Allow 规则 → BYPASS 兜底 → 默认 ASK（或 DONT_ASK 下变 DENY）。

实战：给 Agent 安上权限锁

思路是这样：三组工具（读取、写入、删除），配置成读取自动放行、删除永久禁止、写入要和人确认。
