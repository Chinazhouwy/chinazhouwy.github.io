## 第52题 · 分布式 · 分布式 ID 方案：雪花算法、号段、Redis、自增 ID 怎么选？

**题目**：分布式 ID 方案：雪花算法、号段、Redis、自增 ID 怎么选？

### 用户回答

> 唯一性主键的产生方式，雪花算法其实就是一些固定的规则 + system.current() 生成；redis 用的是自增主键；id 是数据库的主键。

**得分：4/10**

扣分点：
- 雪花算法只说了"固定规则 + System.currentTimeMillis()"，没说清时间戳+机器ID+序列号的具体位分配（-3）
- Redis 和数据库自增只说了结论，没有展开位数限制、性能瓶颈、扩展性问题（-2）
- 没提到 UUID、号段模式、时钟回拨等关键点（-1）

### 追问与补充

用户追问后补充：
- 雪花算法本质 = 时间戳 + 机器唯一标识（+ 序列号防同一毫秒重复）
- 时钟回拨处理 = 业务唯一性校验兜底 + 预分配ID池，不要回头生成

### 最终修正版

#### 六种方案对比

| 方案 | 位数/结构 | 全局唯一 | 有序 | 性能 | 扩展性 | 缺点 |
|------|----------|---------|------|------|--------|------|
| **UUID** | 128位随机 | ✓ | ✗ | 极高 | 无限制 | 无序、索引不友好、占空间 |
| **数据库自增** | bigint 递增 | ✓ | ✓ | 中等 | 受单表限制 | 单点瓶颈、扩展麻烦 |
| **号段模式** | 预取号段+本地分配 | ✓ | ✓ | 高 | 可多节点 | 需要持久化号段 |
| **Redis INCR** | 单线程原子递增 | ✓ | ✓ | 高 | 需集群 | 依赖Redis、持久化风险 |
| **雪花算法** | 64位=时间戳+机器ID+序列号 | ✓ | ✓ | 高 | 可水平扩展 | 时钟回拨问题 |
| **Leaf/美团** | 号段+雪花双模式 | ✓ | ✓ | 高 | 可水平扩展 | 需要部署额外服务 |

#### 雪花算法位分配

```
64位 long 型整数：

| 1 bit 符号位 | 41 bit 时间戳 | 10 bit 机器ID | 12 bit 序列号 |
|    0         | 毫秒级       | 1024台机器    | 每毫秒4096个 |

总容量：约 69 年 × 1024 台 × 4096个/ms ≈ 万亿级/天
```

- **41位时间戳**：相对时间（当前时间 - 起始时间），可用约69年
- **10位机器ID**：5位机房 + 5位机器（可自定义分法）
- **12位序列号**：同一毫秒内可生成 4096 个 ID

#### 时钟回拨处理

```
❌ 错误做法：强行把时间戳设回上一个值
   current = lastTimestamp;  // 强制回退
   → 导致同一毫秒内生成的ID和之前重复

✓ 正确做法：
   1. 业务层加唯一性校验兜底
   2. 预分配ID池到内存，用完异步加载下一批
   3. 不回退时间戳，而是换个机器ID重新生成
```

#### 号段模式（美团 Leaf）

```
数据库表：
┌────┬──────────┬──────────┬───────────┐
│ id │ max_id   │ step     │ biz_tag   │
├────┼──────────┼──────────┼───────────┤
│ 1  │ 1000     │ 1000     │ order     │
└────┴──────────┴──────────┴───────────┘

流程：
1. 应用启动：从DB加载 max_id=1000, step=1000
2. 本地分配：[1001, 2000] 的ID在内存中直接分配
3. 用到一半时：异步预加载下一个号段 [2001, 3000]
4. 用完后：切换到新号段
```

#### 选型决策树

```
需要全局唯一ID
    │
    ├─ 性能要求高 + 可水平扩展？
    │   └─ 是 → 雪花算法（自研或Leaf）
    │
    ├─ 有Redis集群？
    │   └─ 是 → Redis INCR（简单场景）
    │
    ├─ 号码需要连续/可预测？
    │   └─ 是 → 号段模式
    │
    └─ 简单场景、单库？
        └─ 数据库自增
```

### 复习骨架

- 六种方案：UUID / 数据库自增 / 号段模式 / Redis INCR / 雪花算法 / Leaf
- 雪花算法 = 41位时间戳 + 10位机器ID + 12位序列号
- 号段模式 = 预取DB号段 + 本地内存分配 + 双Buffer异步预加载
- 时钟回拨处理 = 业务唯一性校验 + ID池预分配，不回退时间戳
- 选型看需求：有序+高性能→雪花，连续可追溯→号段，简单→DB自增

### R1 回顾（2026-07-09）

**得分：5/10**（R0→R1 +1分）

**答对的部分：**
- 雪花包含时间戳和机器信息 ✓
- Redis 是自增序列 ✓
- 数据库自增消耗数据库性能 ✓
- 雪花是本机生成，不依赖外部 ✓
- 雪花会乱序 ✓

**扣分点：**
- "跟机器地理位置有关" 不准确 → 是 workerId（机器 ID），不是地理位置（-1）
- 雪花位分配没说清 → 41位时间戳+10位机器ID+12位序列号（-2）
- 号段模式原理模糊 → 没说到从DB批量取号、本地分配、用完再取（-1）
- 性能对比有误 → 号段模式性能也很高，不是最慢的（-1）

**关键理解（用户追问后确认）：**
- 雪花"趋势递增"的核心原因：位分配是 时间戳 > 机器ID > 序列号
- 同一毫秒同一台机器 → 严格递增 ✓
- 不同机器 workerId 不同 → 排序乱了 ✗
- 所以多机场景只能保证趋势递增，不能保证严格有序
