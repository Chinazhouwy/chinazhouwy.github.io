## 一、Agent 核心架构

### 四层架构

```
┌─────────────────────────────────────────────────────────┐
│                      感知层（Perception）                 │
│  用户输入 → 解析意图 → 提取关键信息 → 构建上下文           │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                      推理层（Reasoning）                 │
│  LLM推理 → 决策（继续/调用工具/结束）→ 生成行动指令       │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                      行动层（Action）                    │
│  执行工具 → 获取结果 → 更新状态 → 判断是否需要继续循环    │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                      记忆层（Memory）                    │
│  短期记忆（对话上下文）→ 中期记忆（会话摘要）→ 长期记忆（持久化）│
└─────────────────────────────────────────────────────────┘
```

### 核心循环（ReAct Loop）

```
while not done:
    1. 感知：构建当前上下文（用户问题 + 工具描述 + 历史记忆）
    2. 推理：LLM决定下一步（Thought → Action → Observation）
    3. 行动：执行工具，获取结果
    4. 记忆：更新短期记忆，判断是否需要压缩
    5. 判断：任务完成？→ 输出最终答案；未完成？→ 继续循环
```

## 二、ReAct 模式工程化

### ReAct vs CoT 的区别

| 维度 | CoT（Chain of Thought） | ReAct（Reasoning + Acting） |
|------|------------------------|---------------------------|
| **本质** | 模型底层的思维链推理 | 工程化的推理+行动循环 |
| **能力** | 纯推理，不调用外部工具 | 推理+工具调用 |
| **执行** | 一次生成完整推理过程 | 多轮循环，每轮可调工具 |
| **适用** | 简单问答、数学推理 | 复杂任务、需要外部信息 |
| **工程化** | 不需要特殊实现 | 需要Agent框架（LangChain/LlamaIndex） |

## 三、Agent 记忆系统

### 三层记忆

| 层级 | 存储位置 | 生命周期 | 压缩策略 |
|------|----------|----------|----------|
| **短期记忆** | LLM上下文窗口 | 当前对话 | 滑动窗口/摘要压缩 |
| **中期记忆** | 会话存储（Redis/DB） | 当前会话 | 内容摘要/关键信息提取 |
| **长期记忆** | 向量数据库/知识库 | 永久 | 向量化存储/定期清理 |

### 记忆压缩策略

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| **滑动窗口** | 保留最近N条记录 | 对话较短、上下文要求不高 |
| **摘要压缩** | 用LLM总结历史对话 | 长对话、需要保留关键信息 |
| **重要性评分** | 对每条记录打分，保留高分的 | 需要区分信息重要程度 |
| **混合策略** | 最近的保留 + 远的压缩 | 通用场景 |

### 压缩触发条件

| 条件 | 处理方式 |
|------|----------|
| Token数 > 上下文窗口的80% | 触发压缩 |
| 对话轮次 > 阈值（如20轮） | 触发压缩 |
| 用户明确说"总结一下" | 手动触发压缩 |
| 切换话题 | 保存当前上下文，清空短期记忆 |

## 四、多 Agent 协作架构

### 三种模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **主从模式** | 主Agent调度，子Agent执行 | 任务分解型（如：主Agent分析需求，子Agent分别搜索/计算/写代码） |
| **对等模式** | Agent之间平等协作，投票决策 | 需要多视角判断（如：多个Agent分别评审代码，投票决定是否合并） |
| **层级模式** | 多级管理者，层层分发 | 大型项目（如：CEO Agent → 部门Agent → 执行Agent） |

## 五、Agent 可靠性保障

### 幻觉治理

| 手段 | 说明 | 适用场景 |
|------|------|----------|
| **RAG** | 用检索到的事实约束生成 | 知识问答 |
| **事实校验** | 生成后用另一个LLM校验 | 关键信息确认 |
| **置信度评分** | 对生成内容打分，低分不输出 | 高风险场景 |
| **人工审核** | 关键决策前加入人工确认 | 金融/医疗 |
| **引用追溯** | 要求标注信息来源 | 学术/法律 |

### 错误恢复

**错误恢复 = Agent执行任务时出错了，怎么自动补救，而不是直接崩掉。**

| 策略 | 做法 | 例子 |
|------|------|------|
| **重试** | 再试一次（可能只是临时故障） | API超时 → 等2秒再试 |
| **降级** | 换个更简单的方案 | 主模型不行 → 用备用模型 |
| **回滚** | 撤销上一步，换个路走 | A方案失败 → 回退 → 试B方案 |
| **报错** | 实在搞不定，告诉用户 | 3次重试都失败 → "抱歉，目前无法完成" |

**具体场景**：
```
场景1：工具调用失败
  天气API超时 → 自动重试 → 还是超时 → 换一个API → 成功

场景2：LLM返回格式错误
  解析失败 → 让LLM重新生成 → 还是格式错 → 正则提取关键信息 → 手动构造参数

场景3：任务执行到一半发现走不通
  航班已售罄 → 回退一步 → 搜索其他航班 → 找到备选 → 预订成功
```

### 评估指标

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **任务完成率** | 成功完成的任务比例 | >90% |
| **幻觉率** | 生成虚假信息的比例 | <5% |
| **工具调用准确率** | 正确选择工具的比例 | >85% |
| **平均循环次数** | 完成任务需要的平均循环数 | <8次 |

## 六、Function Calling vs Tool Use

| 维度 | Function Calling | Tool Use |
|------|-----------------|----------|
| **定义** | 模型/API 输出结构化函数名和参数的一种机制 | 模型使用外部工具完成任务的泛称 |
| **控制方** | 可以由模型选择函数，也可以由开发者指定函数 | 通常由 Agent 循环结合模型判断、策略和权限控制 |
| **调用时机** | 取决于 API 配置和业务编排，不一定固定 | 取决于任务状态、模型判断和系统策略 |
| **适用场景** | 结构化调用业务函数、查询接口、执行确定动作 | 多步任务、开放域任务、需要观察结果后继续决策 |
| **和 Workflow 的关系** | Function Calling 不是流程编排本身，只是工具调用接口 | Tool Use 也不等于完全自主，仍需要工程侧约束 |

```
Function Calling（结构化接口）：
  用户请求 → 模型输出 {name, arguments} → 程序校验参数 → 调用函数 → 把结果返回给模型或业务
  （它解决“怎么把模型输出变成可执行调用”，不等同于“流程固定”）

Tool Use（Agent驱动）：
  用户请求 → Agent推理 → 决定调用工具X → 获取结果 → 继续推理 → 决定调用工具Y
  （模型自主决定调用什么、何时调用）
```

## 七、用户追问+纠正记录

### 追问1：错误恢复是什么意思？
**用户问题**：不太理解错误恢复的概念
**解释**：Agent执行任务时出错了，怎么自动补救而不是直接崩掉。四种策略：重试、降级、回滚、报错。
**具体场景**：
- 工具调用失败 → 重试或换工具
- LLM返回格式错误 → 重新生成或正则提取
- 任务走不通 → 回退换路

## 八、最终结论

### 骨架答案（一句话核心）
> Agent = 感知→推理/规划→行动→观察→记忆的循环。ReAct 是把推理、工具调用、观察结果串成闭环的工程模式；CoT 更像单次模型调用内的推理提示/表达方式，生产中不一定暴露完整思维链。记忆分三层（短期/中期/长期），压缩策略有滑动窗口、摘要、重要性评分。多 Agent 有主从/对等/层级三种模式。可靠性靠 RAG、工具校验、事实来源、错误恢复（重试/降级/回滚）和业务兜底。

---

## 九、这次讨论的收获

- Agent核心是四层循环：感知→推理→行动→记忆
- ReAct vs CoT：CoT是模型能力，ReAct是工程实现
- 记忆压缩：不能无限增长，要有策略（摘要/滑动窗口/重要性评分）
- 错误恢复：让Agent像人一样遇到问题想办法，而不是直接放弃
- Function Calling 是结构化工具调用机制；Tool Use 是更宽泛的工具使用行为；Workflow/流程编排是开发者定义控制流
