## 详细面试题

### 1. 🚚 海量实时轨迹数据处理

顺丰每天有数百万快递在途，每个快递每分钟上报 GPS 位置。请设计一个轨迹数据处理系统，要求：
1. 实时计算快递预计到达时间（ETA）
2. 支持历史轨迹回放查询（查某快递过去24小时所有点位）
3. 当快递偏离预定路线时10分钟内告警

<details>
<summary>💡 考点提示</summary>

- 流处理（Kafka + Flink）实现实时 ETA 计算
- 时序数据库（InfluxDB/TDengine/Cassandra）存储海量轨迹
- GeoHash 空间索引 + 电子围栏实现偏离告警
- Redis 缓存热轨迹数据，冷数据下沉到 HBase/OSS
</details>

### 2. 📍 智能调度与路径优化

现有 **1000个待派送包裹** 分散在城市各处，**50名快递员** 可用。请设计一个实时调度系统，动态分配包裹给快递员，并规划最优取送路径。

考虑因素：
- 快递员实时位置
- 包裹重量体积
- 收件时间窗
- 交通路况
- 快递员技能差异（如某些只能送文件）

<details>
<summary>💡 考点提示</summary>

- 车辆路径问题（VRPTW）变种 → 使用 OR-Tools/LKH 求解
- 实时调度引擎 + 增量重优化（插入/移除包裹）
- 约束求解器（时间窗、容量、技能匹配）
- 地理围栏 + 实时路况 API 动态更新路径
</details>

### 3. 📦 仓储作业并发控制

双11期间，某仓库有 **100个拣货员** 同时作业。当多个拣货员通过 PDA 扫描同一货架的同一商品时，如何防止超拣（库存充足但多人争抢导致有人失败）？

请设计从 PDA 到 WMS 系统的完整方案，考虑网络抖动、设备异常等情况。

<details>
<summary>💡 考点提示</summary>

- 分布式锁（Redis RedLock / ZooKeeper）
- 乐观锁（CAS + 版本号）降低锁竞争
- 库存预占 + 预扣 + 异步释放
- 本地降级（PDA离线容灾，本地暂存队列）
- 最终一致性 + 对账补偿
</details>

### 4. 🔄 运单状态机一致性

快递状态有：已下单、已揽收、运输中、到达网点、派送中、已签收、异常等 **20+ 状态**。状态变更可能来自快递员 APP、分拣机、客户端等不同系统。

如何保证：
1. 状态流转符合业务规则（如不能从"运输中"直接到"已签收"）
2. 多个系统同时修改状态时不冲突
3. 客户看到的最终状态一致

<details>
<summary>💡 考点提示</summary>

- 有限状态机（FSM）+ DSL 规则引擎（Drools/状态表）
- 版本号乐观锁/CAS 控制并发冲突
- 事件溯源（Event Sourcing）记录完整状态变更历史
- CQRS 读写分离：写端保证有序，读端最终一致
- 基于本地消息表 + MQ 保证跨系统状态一致性
</details>

### 5. 💰 运费实时计算引擎

顺丰运费计算涉及：重量、体积、距离、时效（当日达/隔日达）、包装、保价、节假日附加费等 **20+ 因子**，且有 **500+ 条特殊计费规则**（如某大客户打8折）。

请设计一个运费计算服务，要求：
- 计算耗时 < 50ms
- 支持规则热更新
- 能记录每笔运费的计算明细用于对账

<details>
<summary>💡 考点提示</summary>

- 规则引擎（Drools / Aviator / 自研 DSL）
- 表达式缓存 + 计算缓存（计算因子不变时复用结果）
- 规则热更新：配置中心（Nacos/Apollo）+ 版本管理
- 计算快照日志 → 写明细表或 ES 用于对账追溯
- 计费因子结构化（策略模式 + SPI）
</details>

### 6. 🕐 预约配送时间窗管理

客户可预约"明天14:00-16:00"上门取件，每个时间窗容量有限。请设计预约系统，防止同一时间窗被超额预约。

需要考虑：
1. 不同网点容量不同
2. 快递员临时请假需释放容量
3. 预约后修改时间窗

<details>
<summary>💡 考点提示</summary>

- Redis 原子自增 + Lua 脚本控制容量扣减
- 分布式锁防止超售（分段锁，按网点+时间窗 hash）
- 时间窗容量表 → 预扣 → 确认 → 释放（TCC 模式）
- 请假事件 → MQ 异步释放对应时间窗容量
- 预约修改 = 释放旧窗 + 锁定新窗（事务）
</details>

### 7. 📋 多主体自动对账系统

一笔快递交易涉及：寄件人、收件人、顺丰、快递员、第三方保险公司等 **多方资金结算**。请设计日终自动对账系统，能快速定位不平账单（如寄件人已付运费但系统未记录）。

要求：
- 对账 10万笔/分钟
- 次日6点前完成昨日全量对账

<details>
<summary>💡 考点提示</summary>

- 对账引擎：各方交易流水按唯一ID（运单号+交易流水号） join
- 分片策略（按网点/时间/运单号 hash）并行对账
- 对账状态机：待对账 → 一致 / 长款 / 短款 / 差错
- 差错自动修复（长款原路退回、短款补扣）
- 批量 vs 流式：日切后批处理，辅以实时流对账（Flink）
</details>

### 8. 🎟️ 优惠券与营销结算

618期间发放 **1000万张"满30减5"优惠券**，要求：
1. 防止超发
2. 防止一张券重复使用
3. 实时核销（支付时立即抵扣）
4. 日终统计各渠道用券数据给财务

<details>
<summary>💡 考点提示</summary>

- 发券：Redis 预生成券池（lpush）→ 预扣库存 → lpop 发放
- 用券：Redis 记录券状态（已领取/已使用/已过期）
- 核销：分布式锁防止重复消费 + 数据库唯一约束
- 日终：离线批处理 + MQ 异步统计报表
- 兜底：库存最终一致 + 对账补偿
</details>

### 9. 💳 资金安全与防重付

客户在顺丰 APP、微信、官网等多渠道支付同一笔运费，如何防止重复支付？若已重复支付，如何自动原路退款？

请考虑：第三方支付回调延迟、网络超时重试、银行日切等边界情况。

<details>
<summary>💡 考点提示</summary>

- 业务幂等：支付请求 ID（幂等键）+ 交易状态机
- 数据库唯一索引防重
- 重复支付检测 → 自动退款引擎（原路退回）
- 第三方回调：异步对账 + 补偿任务兜底
- 日切处理：银行对账文件 reconcile → 差错处理
</details>

### 11. 📝 工单自动分配与升级

客户投诉自动生成工单，需根据：投诉类型、客户等级、问题紧急度、技能组忙闲度等因素，智能分配给合适客服。若2小时未处理，自动升级给主管。

请设计工单流转引擎，支持可视化配置分配规则。

<details>

### 12. ⚠️ 预测性客诉预警

基于历史数据预测某批快递可能大面积延误，提前主动推送通知给客户，减少客诉。请设计预警系统，从数据收集、特征工程、模型预测到预警触发的完整架构。

<details>
<summary>💡 考点提示</summary>

- 离线特征：历史延误率、天气、路况、节假日
- 实时特征：实时位置、速度、中转停留时长
- 模型：GBDT/XGBoost 或时序预测（LSTM/Prophet）
- 预警策略：风险分 > 阈值 → 主动推送（模板 + 渠道分发）
- AB 实验：预警策略灰度验证，对比客诉率
</details>

### 13. 🤖 仓储机器人调度系统

仓库有 **200台 AGV 搬运机器人**，需接收 WMS 系统的搬运指令，调度机器人到指定货架取货，送至打包台，要求：
1. 防止机器人拥堵碰撞
2. 动态调整路径（如某区域临时维修）
3. 机器人电量低于20%时自动充电

<details>
<summary>💡 考点提示</summary>

- 路径规划：A* / D* Lite 算法（动态避障）
- 交通管制：路段占用锁 + 拓扑图网格
- 充电调度：电量管理 → 空闲时自动回充
- 中央调度 vs 分布式协商
</details>

### 14. 🗄️ 智能快递柜并发控制

快递员在小区快递柜存件时，可能多个快递员同时操作同一柜子。请设计分布式锁方案，保证：
1. 存件不冲突
2. 取件码唯一
3. 快递柜离线时（网络断开）仍能正常工作

<details>
<summary>💡 考点提示</summary>

- 分布式锁（Redis RedLock / etcd） + 租约自动续期
- 取件码：Redis INCR 自增 + 重试去重
- 离线容灾：柜端本地缓存 + 离线模式 → 恢复后同步
- 柜锁状态：本地文件存储 + 同步到云端
</details>

### 18. 🚀 灰度发布与流量调度

新开发智能调度算法，需在深圳5个网点先灰度验证效果。请设计流量调度方案，能将指定网点的请求流量导入新算法，且可快速回滚。

<details>
<summary>💡 考点提示</summary>

- 全链路灰度：每个服务识别 + 透传灰度标签
- 快速回滚：路由表实时切换 + 旧版本保留
- 监控对比：灰度 vs 基线，看板实时对比关键指标
</details>
