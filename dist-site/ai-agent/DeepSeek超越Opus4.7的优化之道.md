# 我们是如何让 DeepSeek 超越 Opus 4.7 的？

> **来源**: 小红书（翻译自 Twitter @MrAhmadAwais）
> **原文链接**: https://x.com/MrAhmadAwais/status/2050956678502420612
> **NoteId**: 6a0bdedf000000003502804d
> **标签**: #AI #DeepSeek #模型优化 #工具调用
> **考点分类**: 模型优化、工具调用、Schema设计、输入修复

---

## 核心观点

> "开放模型不擅长工具调用"几乎总是**驾驭层（Orchestration Layer）**的问题，而非模型本身的问题。

---

## 背景

作者在 **@CommandCodeAI**（一个基于开源 AI 的命令行工具）上花了两天时间，仔细审查了数十亿个 token，使用的是 DeepSeek。最终编写了一个**工具输入修复层（tool-input repair layer）**。

**触发点**: 看到 DeepSeek-Flash 在最简单的 `/review` 运行中失败——每次 `shellCommand` 和 `readFile` 调用都会返回一堆原始的 Zod 校验错误信息，模型无法恢复，因为错误信息不是它能读懂的格式。

**结果**: 到结束时，**DeepSeek v4 Pro 在内部评估中，已经能在 6/10 的测试中击败 Opus 4.7**。

---

## 发现一：失败模式不是随机的——它们是一个很小的、有限的结构化集合

在 DeepSeek-Flash、DeepSeek v4 Pro、GLM、Qwen 中，几乎完全相同的四种错误反复出现：

1. 对可选字段发送 `null`，而不是直接省略它
2. 将 `["a","b"]` 作为一个 JSON 字符串发出，而不是一个真正的数组
3. 将单个参数包裹在 `{}` 中，而 schema 期待的是一个数组（一个"空占位符"）
4. 在期待数组的地方传递了一个裸字符串（`"foo"` 而不是 `["foo"]`）

**四种修复方式**，每种约 30-100 行代码，需要按顺序执行（`json-array-parse` 必须在 `bare-string-wrap` 之前运行，否则 `'[\"a\",\"b\"]'` 会变成 `[['[\"a\",\"b\"]']]`）。这就是全部目录了。

> 当我听到"这个开源模型不会做工具调用"时，我现在默认就是这四种情况之一，到目前为止，约 90% 的情况下是对的。

---

## 发现二：最有趣的失败模式，也是最具有揭示性的

DeepSeek-Flash 在被要求编辑或写入文件时，有时会将路径格式化为 Markdown 自动链接：

```
filePath: "/Users/x/proj/[notes.md](http://notes.md)"
```

writeFile 工具老老实实地尝试创建字面名称的文件，直到发现了这个问题。

**这不是幻觉**。这是**后训练聊天分布泄漏到了工具边界**——模型在对话输出中因自动链接而获得了奖励，所以它将在一种毫无意义的上下文中也应用这一先验知识。

**修复方法**: 两行正则表达式，只处理退化情况（链接文本等于不带协议的 URL），而真正的 Markdown 如 `[click](https://x.com)` 则原封不动地通过。

这也是**工具混淆（tool confusion）**——模型在 RL 阶段使用的自有工具，与我们编写的所有其他工具不同，当然也无法预测。

> "工具混淆"是一个比"能力差距"更有用的框架。模型知道如何格式化路径。

**路径问题的根本原因**: 它只是还没有被足够清晰地告知：这条路径将用于 `fopen`，而不是用于聊天气泡。所以在 schema 层面编码了这个提示——使用 `pathString()` 而不是 `z.string()`——然后这个泄漏问题就对所有路径字段一次性修复了。

---

## 发现三：关键的设计选择——将"先预处理再校验"颠倒为"先校验再修复"

**第一次尝试**（预处理通行）: 在 Zod 看到输入之前先对输入进行规范化（剔除 null、解析字符串化的数组等）。

**问题**: 某个恰好长得像 JSON 的 writeFile 内容，在被写到磁盘之前就被重写了。静默的数据损坏，在冒烟测试中很容易被忽略。

**改进后的方案**:

1. **按原样解析输入**。如果成功了，就直接发送出去。有效的输入永远不会被触碰。
2. **如果失败，遍历校验器自己生成的 issue 列表**。对每个 issue 路径，按顺序尝试四种修复方式，直到一种适用。
3. **再次解析**。如果成功，记录 `tool_input_repaired:${toolName}`。如果失败，记录 `tool_input_invalid:${toolName}`，并返回一条模型可读的重试消息。

**结构性洞见**:

> 当你预处理时，你编码了一个关于"什么坏了"的先验知识。当你先让校验器报错时，**schema 本身就是先验知识**，你只在 schema 实际不认可的精确路径上花费修复成本。校验器正在为你做定位 bug 的工作。

这与"先低成本尝试，再谨慎处理"的模式完全相同——先走快速路径，在有证据时才回退。

> 这也免费为你提供了每个工具的遥测数据。你可以按（模型，工具）维度观察修复率，并在用户发现问题之前，就注意到某个模型在特定合约上发生了退化。

---

## 发现四：形状不变量和关系不变量需要不同的修复方式

上述四种修复方式处理的都是**形状问题**——类型错误、缺少键、容器错误。但 `readFile` 有一个**关系不变量**：

> "如果你提供了 offset，你也必须提供 limit，反之亦然。"

DeepSeek 持续调用 `readFile({ absolutePath, limit: 30 })`，然后收到一条 `ERROR:` 回应。

**你不能通过输入修复来解决这个问题**，因为每个字段单独来看都是有效的——bug 出在字段之间的关系上。

**解决方案**: 教会函数理解模型的意图。
- `limit` 单独出现 → `offset = 0`
- `offset` 单独出现 → `limit = 2000`（匹配常见的 read tool 操作的默认值）

然后在结果中将这个决策反馈给模型：

> "注意：limit 未提供，已默认设为 2000。如需读取更多或更少行，请在重试时同时提供 offset 和 limit。"

**没有 `Error:` 前缀**，所以终端不会把它标红。模型看到了我们的选择，如果我们的猜测不对，它可以在下一轮自行纠正。

**透明胜过静默的魔法。**

> **能修复的地方就修复。不能修复的地方就扩展语义。无论哪种方式，都要将选择透明化。**

---

## 总体视角

很多看起来像是"模型能力"的东西，实际上是**合约设计**的问题。

严格的 schema 是一个带有成本的选择——它过滤掉了噪音，但也过滤掉了来自任何尚未记住你恰好选择的精确 JSON 合约的模型的可恢复噪音。

| 对比维度 | 最大的商业模型 | 开放模型 |
|---------|--------------|---------|
| 训练数据 | 大规模训练，见过足够多的各种合约 | 数据量较少 |
| 工具调用表现 | 默默承担严格合约的成本，表现得宽容 | 高声承担成本，因此被否定 |

**驾驭层是你可以介入调解这些问题的地方**。

四种小的修复（相信后续还会有更多——目前还有三个正在合并中）、两行处理自动链接的正则表达式、一个关系默认值、一个前缀改动。**模型没有变。合约在恰好需要的地方变得更为宽容。**

---

## 结论

> DeepSeek v4 Pro 现在在我们的内部评估中，能在 6/10 的测试中击败 Opus 4.7。

在我看来，"技能问题"更多时候适用于**驾驭层**，而非模型本身。
