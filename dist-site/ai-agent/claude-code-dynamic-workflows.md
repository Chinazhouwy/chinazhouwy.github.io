## 核心概念

Claude Code 的动态工作流（Dynamic Workflows）：**主 Agent 根据当前任务临时生成一个 JavaScript 工作流文件，安排多个子 Agent 分头调查、执行、验证，再把结果收回来。**

不是简单的并发调用，而是**临时搭起来的执行框架（Harness）**：先决定拆成几条线，哪些并行，哪些需要复核，最后由谁收口。

---

## 为什么需要动态工作流

### 单 Agent 的局限
- 短任务靠一个 Agent 一路做完，沟通成本低，状态集中
- **长任务问题**：
  - 上下文容易被塞满（材料越多越容易溢出）
  - 路径依赖（模型沿早期判断继续走）
  - 难以判断哪些结论有证据，哪些是猜测

### 两个关键问题
1. **Agentic Laziness（Agent 懒惰）**：长任务执行到中后段，模型倾向于给出看似完整的收尾，而不是继续追完剩下的分支
2. **Self-preferential Bias（自我偏好）**：模型偏向相信自己前面产出的结果，忽略反向验证

---

## 动态工作流的组成

| 组件 | 通俗理解 | 风险点 |
|------|---------|--------|
| 主 Agent | 项目负责人 | 任务拆错，后面都偏 |
| JavaScript 工作流文件 | 临时施工方案 | 脚本本身要能读懂、能检查 |
| 子 Agent | 专项工程师 | 可能重复劳动或局部最优 |
| 验证/合成步骤 | 复核和交付检查 | 验证标准太弱会放过错误 |

---

## 六种工作流模式

| 模式 | 通俗解释 | 适用场景 |
|------|---------|---------|
| **Classify-And-Act** | 先分类，再按类型走不同处理路径 | Issue 分类（bug/文档/性能/安全） |
| **Fanout-And-Synthesize** | 多方向并行调查，最后汇总 | 大 PR 同时看 API/测试/性能/部署/文档 |
| **Adversarial Verification** | 专门安排反向验证者挑错 | 迁移方案审查、安全结论验证 |
| **Generate-And-Filter** | 先生成多组候选，再按规则过滤 | 生成多套修复方案，按风险/成本筛选 |
| **Tournament** | 多 Agent 做同一任务，互评或裁决 | 多种实现方案竞争，由 judge 选择 |
| **Loop Until Done** | 反复执行直到满足停止条件 | 测试失败持续排查（必须有停止条件） |

**最适合日常开发**：Fanout-And-Synthesize（资料多/分支多）+ Adversarial Verification（结论需复核）

---

## 工作流基元（Helper Functions）

| 基元 | 做什么 | 适合场景 |
|------|--------|---------|
| `agent(prompt, opts?)` | 启动子 Agent，给独立任务/模型/工具 | security reviewer、test planner |
| `parallel([fns])` | 同时运行多个函数/子任务 | 大 PR 审查、多路径排障 |
| `pipeline(items, ...)` | 批量输入按流水线逐步处理 | 批量 issue 分类、逐项验证 |

---

## 与其他方式的区别

| 方式 | 适合什么 | 优点 | 局限 |
|------|---------|------|------|
| 普通 prompt | 一次性小任务 | 快，成本低 | 不稳定，难复用 |
| Slash command | 高频固定动作 | 入口清晰，适合团队约定 | 不适合复杂分支 |
| Subagents | 固定角色分工 | 独立上下文，职责明确 | 需要上层流程协调 |
| **Dynamic workflows** | 当前任务的临时编排 | 能拆分、并行、验证、合成 | 消耗更多 token，流程质量要检查 |

**动态工作流的价值在于组织已有能力**：项目里定义常用 subagents（reviewer/security-auditor/test-runner），复杂任务中动态工作流决定调用哪些角色、按什么顺序、结果怎么交叉验证。

---

## 适合用动态工作流的任务

| 任务 | 为什么适合 | 验收方式 |
|------|-----------|---------|
| 大 PR 代码审查 | 可拆成 API/测试/性能/文档/安全 | 按风险等级排序的审查报告 |
| 失败测试排查 | 不同子 Agent 分别看日志/代码/提交 | 可能原因 + 可复现实验 |
| 安全配置检查 | 分开看权限/依赖/网络/密钥 | 风险列表 + 修复建议 |
| 技术调研 | 分别查文档/示例/开源实践 | 带来源的对比表 |
| 跨模块重构评估 | 拆成影响范围/迁移步骤/测试策略 | 迁移计划 + 回滚方案 |

---

## 审计要点（必查）

1. **拆分是否合理**：子任务之间有没有重复、遗漏和依赖关系
2. **证据是否可追溯**：每个结论能不能回到文件、命令、文档或日志
3. **合成是否处理冲突**：不同子 Agent 结论不一致时谁来裁决

> **多 Agent 最大风险是把冲突包装成确定结论**
