## 第 11 题 · MySQL · 联合索引设计

**题目**：联合索引 (a, b, c)，以下哪些查询能用到索引？为什么？

1. `WHERE a=1 AND b=2 AND c=3`
2. `WHERE a=1 AND c=3`
3. `WHERE b=2 AND c=3`
4. `WHERE a=1 AND b>2 AND c=3`
5. `WHERE a=1 ORDER BY b`

追问：什么是索引下推（ICP）？MySQL 5.6和8.0有什么区别？

### 用户回答（摘要）

> 1和4。剩下不清楚。

### 评分：2/10

### 扣分点

1. **第4条理解有误**（-2）：b>2 是范围查询，c 不能继续用于索引范围定位，但可能通过 ICP 在索引层过滤
2. 第2、3、5条都说不清楚（-3）

### 逐条分析

**① `WHERE a=1 AND b=2 AND c=3`** → ✅ 全部用到
```
a=1 精确匹配 → b=2 精确匹配 → c=3 精确匹配
三个列都走索引，效率最高
```

**② `WHERE a=1 AND c=3`** → ⚠️ 能用到a，c取决于MySQL版本
```
a=1 ✅ 用到索引
b   ❌ 跳过了，索引断裂
c=3 → MySQL 5.6之前：全表扫描c
     → MySQL 5.6+（有索引下推ICP）：在索引层面过滤c，不用回表再过滤
```

**③ `WHERE b=2 AND c=3`** → ❌ 通常不能用联合索引做定位
```
没有a，直接跳到b → 违反最左前缀 → 通常全表扫描或索引全扫描；MySQL 8.0 在特定数据分布下可能使用 skip scan，但面试主结论仍是“不满足最左前缀”
```

**④ `WHERE a=1 AND b>2 AND c=3`** → ⚠️ 只能用到a和b
```
a=1 ✅ 精确匹配
b>2 ✅ 范围查询，用到索引
c=3 ⚠️ b是范围查询后，c不能继续缩小索引扫描范围；如果开启 ICP，c 仍可能在索引层过滤，减少回表
```

**⑤ `WHERE a=1 ORDER BY b`** → ✅ 能用到索引
```
a=1 ✅ 用索引定位
ORDER BY b ✅ a=1的情况下，b在索引中已排好序
→ 不需要额外的filesort
```

### 核心原则：最左前缀匹配

```
联合索引 (a, b, c) 的B+树结构：

        [a=1]
       /  |  \
   [b=1] [b=2] [b=3]
    /|    /|    /|
 c=1 c=2 c=1 c=2 c=1 c=2

索引是按 a → b → c 的顺序排列的
跳过前面的列，后面的列用不到索引
```

### 能用到索引的组合总结

| 查询条件 | 能用到索引的列 | 说明 |
|---------|-------------|------|
| `a=1 AND b=2 AND c=3` | a, b, c | 全部用到 |
| `a=1 AND b=2` | a, b | 用到前两列 |
| `a=1` | a | 只用第一列 |
| `a=1 AND c=3` | a（主）+ c（ICP） | 跳过了b |
| `a=1 AND b>2 AND c=3` | a + b（范围）+ c（可能 ICP 过滤） | c 不能继续做范围定位 |
| `b=2 AND c=3` | 通常不能做索引定位 | 没有a；MySQL 8.0 特定场景可能 skip scan |
| `a=1 ORDER BY b` | a + b排序 | 利用索引有序性 |

### 追问：跳过中间列会怎样？

```sql
WHERE a=1 AND c=3
```

索引能用，但打折扣：
- a=1 能通过索引快速定位
- c=3 跳过了b，c的值在a=1的数据中是无序的
- 只能逐行过滤c（ICP可以在索引层过滤，减少回表）

**真正"用到联合索引"的场景是不跳过的情况。跳过中间列只是"碰巧用到了前缀"。**

### 建索引的原则

```
查询：WHERE a=1 AND c=3

错误建法：(a, b, c)  → b用不到，浪费空间和写入性能
正确建法：(a, c)     → a和c都用到，索引利用率100%
```

### 索引下推（ICP = Index Condition Pushdown）

**本质：把"回表后过滤"提前到"索引层过滤"，减少回表次数。**

**MySQL 5.6之前：**
```
WHERE a=1 AND c=3
1. 索引找到 a=1 的记录 → 拿到主键
2. 回表（到聚簇索引取完整行数据）
3. 在Server层过滤 c=3
→ 回表了很多不需要的数据
```

**MySQL 5.6+（有ICP）：**
```
WHERE a=1 AND c=3
1. 索引找到 a=1 的记录
2. 在索引层直接检查 c=3（因为c也在索引里）
3. 只有同时满足 a=1 AND c=3 的才回表
→ 减少了回表次数
```

**EXPLAIN中看ICP：** Extra列出现 "Using index condition" → 说明走了ICP

### ICP不限于联合索引

单列索引也能用ICP，关键是被过滤的列要包含在某个索引里。

```
联合索引 (a, b, c)：WHERE a=1 AND c=3 → ICP能帮c过滤 ✅
单列索引 idx_a(a)：WHERE a=1 AND c=3 → ICP帮不了c ❌
联合索引 (a, c)：WHERE a=1 AND c=3 → ICP能帮c过滤 ✅
```

### 速记口诀

> **联合索引从左往右用，跳过的列后面全作废；范围查询后面的列也作废；排序的列跟在等值条件后面能用索引。ICP = 把过滤下推到索引层，减少回表。**
