## 一、核心概念（带场景例子）

### 1. 缓存穿透（Cache Penetration）—— 专挑不存在的路走
请求的key在数据库里也不存在，每次miss都打到DB。
- 场景：黑客用脚本疯狂请求不存在的ID
- 解法：布隆过滤器前置拦截 + 缓存空值

### 2. 缓存击穿（Cache Breakdown）—— 集中攻击一个热点
单个热点key突然失效，大量并发请求同时打到数据库。
- 场景：大促时明星同款商品缓存过期
- 解法：互斥锁 + 双重检查、逻辑过期

### 3. 缓存雪崩（Cache Avalanche）—— 大面积轰炸
大量key同时失效或Redis宕机，所有请求涌向DB。
- 场景：批量导入数据设置了相同过期时间
- 解法：随机过期时间 + 多级缓存 + 集群 + 熔断降级

## 二、三种问题对比速查表

| 维度 | 缓存穿透 | 缓存击穿 | 缓存雪崩 |
|------|---------|---------|---------|
| 攻击对象 | 不存在的key | 单个热点key | 大量key / Redis整体 |
| 成因 | 查询DB也为空 | 热点key突然过期 | 大面积同时失效 / Redis宕机 |
| 并发量 | 可以很高（恶意刷） | 高并发打一个点 | 极高（全量请求） |
| 后果 | DB压力持续增大 | 热点数据重建期间DB压力大 | DB直接被挂 |
| 核心解法 | 布隆过滤器 + 缓存空值 | 互斥锁 + 逻辑过期 | 随机过期 + 集群 + 熔断 |

## 三、解决方案

### 布隆过滤器（穿透）
- Redisson BloomFilter，预估数据量 + 误判率初始化
- 写入时同步写BF，查询时先查BF再查Redis/DB
- BF返回"不存在"→ 一定不存在，直接返回null

### 互斥锁（击穿）
- Redisson分布式锁 + 双重检查
- 只有一个线程重建缓存，其他等待

### 随机过期时间（雪崩）
- baseTTL + random(0, 300s)，避免同时失效
- 多级缓存：本地缓存(Caffeine 1s) → Redis(5~10min) → 熔断降级

### 逻辑过期（击穿/雪崩的进阶方案）
- 不设物理TTL，value里存expireTime
- 过期时异步刷新，当前请求返回旧数据
- 代价：短暂不一致，适合读多写少场景

### 延迟双删（数据一致性）
- 更新DB → 删缓存 → 等500ms → 再删一次
- 解决并发读线程把旧数据写回缓存的问题
- **不完美**：延迟时间是经验值，不能保证100%一致
- **真相**：缓存和DB的强一致性，在不引入分布式事务的前提下不可能
- 金融扣款等强一致场景不能用缓存，直接走DB

## 四、追问记录

### 追问1：布隆过滤器误判
- 不存在的key会误判为存在（假阳性），存在的key不会误判为不存在
- 误判只是第一道防线，后续还有Redis和DB验证
- 实际影响可接受：最多多查一次DB返回空，然后写入缓存

### 追问2：逻辑过期的数据一致性
- 版本号方案：每次读都要查DB，缓存形同虚设，不推荐
- 标准做法：逻辑过期 + 异步刷新，用短暂不一致换高可用
- Canal监听binlog：更高级但引入中间件，大多数项目不需要

### 追问3：延迟双删真的有效吗？
- 核心问题：500ms是经验值，如果并发读线程更慢，第二次删也白删
- **没有银弹**：没有分布式事务就不可能强一致
- 工程取舍：大部分业务容忍几秒不一致，延迟双删+随机TTL够用

## 五、面试回答模板

> 缓存穿透、击穿、雪崩是三种不同的缓存失效场景。穿透是查询不存在的数据，用布隆过滤器+缓存空值；击穿是单个热点key过期，用互斥锁或逻辑过期；雪崩是大面积key同时失效或Redis宕机，用随机过期时间+多级缓存+熔断降级兜底。数据一致性方面，大部分场景延迟双删+随机TTL够用，强一致场景不能用缓存直接走DB。没有银弹，关键是根据业务场景做取舍。
