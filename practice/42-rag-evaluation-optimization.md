## 核心概念

### 三个核心指标（一句话版本）

```
召回率 = 该找的 → 找全了没？
精确率 = 找到的 → 准不准？
幻觉率 = 回答的 → 有没有编？
```

### 召回率

```
召回率 = 检索到的相关文档 / 所有相关文档

举例：10篇相关文章，检索返回5篇，4篇对的
召回率 = 4/10 = 40%（漏了一半）

召回率低 = 漏了东西
```

### 精确率

```
精确率 = 检索到的相关文档 / 检索返回的文档总数

举例：返回5篇，4篇对的，1篇无关
精确率 = 4/5 = 80%

精确率低 = 垃圾太多
```

### 幻觉率

```
幻觉 = 文档没说，模型自己编的

举例：文档只说"主从切换会丢锁"
模型回答"建议用RedLock解决"（文档没提RedLock）
→ 这就是幻觉

幻觉率 = 编造内容 / 总输出内容
```

### 精排指标

```
MRR = 第一个相关结果的排名倒数
  排序 [无关, 无关, 相关] → MRR = 1/3 = 0.33
  排序 [相关, 无关, 无关] → MRR = 1/1 = 1.0

NDCG = 考虑排序质量的指标，相关度高的排前面得分高
```

### 生产中真正在用的评测方式

```
① 人工评测（最直接，但成本高）
② 用户反馈（👍👎，成本为零，数据量大）
③ A/B 测试（改策略灰度10%流量，看业务结果）
④ 黑盒评测（50个高频问题，人工打分1-5分）

学术指标（Recall/NDCG/幻觉率）更多是：
  调参时对比用
  写论文/汇报时量化用
  大厂有专人做评测体系时才用
```

### LLM-as-Judge（用模型评模型）

```
定位：验证方案，不是生产方案
用法：改了检索策略/模型后，跑一批测试集验证效果
做法：把 [检索文档] + [生成回答] 喂给检查模型，判断是否有依据

能判断的：精确率、幻觉率（看结果对不对）
判断不了的：召回率（需要知道"全集"，模型看不到没检索到的文档）
```

### RAG 优化方向

```
召回率低 → 加BM25混合检索（向量+关键词）
精确率低 → 加Reranker精排过滤
幻觉率高 → Prompt约束"只根据文档回答"+ 引用标记
延迟高 → 缓存热门查询 + 向量索引优化
```

### 混合检索（提升召回率的核心方案）

```
向量检索 = 找"意思相近"的（语义匹配）
BM25 = 找"关键词一样"的（精确匹配）
混合 = 两边都不漏 → 召回率最高

融合方式：
  RRF：score = 1/(k+rank_向量) + 1/(k+rank_BM25)，k=60
  加权：α×向量得分 + (1-α)×BM25得分，α=0.6~0.7
```

## 用户追问纠正记录

1. 人工评测是最真实的，但成本高、不可自动化
2. LLM-as-Judge是验证方案不是生产方案，用户判断正确
3. 召回率需要知道"全集"，小模型判断不了，只能离线测
4. BM25（稀疏模型/倒排索引）是提升召回率的关键手段
5. 这些学术指标在生产中大部分团队不用，用户直觉正确

---

## 这次讨论的收获

- 召回率=找全了没，精确率=准不准，幻觉率=有没有编
- 生产中评测主要靠用户反馈（👍👎）和人工抽检，不是学术指标
- LLM-as-Judge是验证方案，判断精确率和幻觉率可以，判断召回率不行
- 提升召回率用混合检索：向量+BM25
- 面试答"用户反馈+A/B测试"比背指标更真实

## GPT 纠错

- GPT 纠错：幻觉率不能简单定义为“文档没说的内容占总输出内容”。RAG 更常拆分为 faithfulness/groundedness（回答是否被上下文支持）与 factual correctness（事实是否正确）。
- GPT 纠错：LLM-as-Judge 不只是验证阶段方案，也可用于离线回归和线上抽样监控，但必须校准偏差、稳定性和评判标准。
- GPT 纠错：有参考答案、相关文档标注或完整语料集合时，LLM 可以参与评估 context recall；不能绝对说它“判断不了召回率”。
- GPT 纠错：点赞点踩并非零成本且存在选择偏差，不能替代成体系的离线测试集、人工标注和业务指标。
- GPT 纠错：Recall、MRR、NDCG、faithfulness 等指标在生产 RAG 中并非只用于论文，它们常用于版本回归和检索策略对比。
