## 第7题 · 并发编程 · lock() vs tryLock() 在 AQS 层面的区别

**题目**：ReentrantLock 的 `lock()` 和 `tryLock()` 有什么区别？在 AQS 层面，两者的执行流程有什么不同？追问：tryLock() 失败会进 CLH 队列吗？

### 用户回答（摘要）

> lock 是直接 cas 抢，失败了就进入 CLH 队列

### 评分：3/10

### 扣分点

1. tryLock() 完全没提（-4）
2. lock() 流程只说了 CAS 抢，漏了可重入判断和排队逻辑（-2）
3. 追问没答：tryLock() 失败是否进队列（-1）

### 正确答案

| | lock() | tryLock() |
|--|--------|-----------|
| 阻塞 | ✅ 阻塞，拿不到死等 | ❌ 非阻塞，立刻返回 false |
| 进队列 | 失败后进 CLH 队列，park() 睡眠 | 失败直接返回，不进队列 |
| 可中断 | lock() 不可中断 | tryLock(timeout) 可超时 |
| 适用场景 | 必须拿到锁 | 拿不到就做别的事 |

**一句话**：lock() 拿不到就排队睡觉，tryLock() 拿不到就立刻走人。

## 回顾记录（2026-07-02，R5）

**得分：5/10**（首次 3/10，进步 2 分）

### 用户回答
- AQS 维护双向队列（CLH），带哨兵头节点
- 拿不到锁的线程排到队尾，释放锁从队头取一个唤醒
- state 变量控制锁，表示锁能不能拿到
- 可重入锁利用 state 判定重入次数
- 共享 vs 独占 = 数据是否可以有多个值，不只是 0 和 1

### 追问+纠正记录
1. **CLH 队列细节**：每个 Node 存 thread、waitStatus、prev/next 指针；Head 是哨兵节点不存线程，真正等锁的从 Head.next 开始
2. **state 变量含义**：取决于锁类型——独占锁 state=0 空闲/1 占用/N 重入次数；共享锁 state=剩余许可数
3. **共享 vs 独占纠正**：不是"数据多个值"，独占=同一时刻 1 个线程能获取，共享=允许多线程同时获取（如 Semaphore）
4. **ReadWriteLock**：state 高 16 位管读锁计数，低 16 位管写锁计数，一个变量同时管两种锁

### 最终结论
AQS 核心是 CLH 双向队列 + volatile int state。拿不到锁的线程封装成 Node 排队，释放锁时从队头唤醒。独占模式 state 表示锁占用状态，共享模式 state 表示剩余许可数。

### 这次讨论的收获
- CLH 队列是双向链表，有哨兵头节点
- state 不是简单的 0/1，在不同锁类型中含义不同
- ReadWriteLock 用一个 state 高低位同时管理读写锁
