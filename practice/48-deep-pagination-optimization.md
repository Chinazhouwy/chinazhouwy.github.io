## 第48题 · MySQL · 深分页优化

**题目**：深分页怎么优化？覆盖索引、延迟关联、游标分页怎么选？

## 一、LIMIT 1000000, 10 为什么慢？

```sql
SELECT * FROM orders WHERE status = 1 ORDER BY id LIMIT 1000000, 10;
```

MySQL 执行过程：
```
1. 走索引找到 status=1 的第一条记录
2. 沿着索引往后扫 1000010 行（不能跳过！）
3. 前 1000000 行全部丢弃
4. 只取最后 10 行
5. 这 10 行还要回表取 *（所有字段）
```

**瓶颈在两步**：
- 第 2 步：白扫了 100 万行（CPU + IO）
- 第 5 步：10 行要回表 10 次（随机 IO）

## 二、四种优化方案

### 方案1：覆盖索引（最简单）

```sql
-- 原来：select * 需要回表
SELECT * FROM orders WHERE status = 1 ORDER BY id LIMIT 1000000, 10;

-- 优化：只查索引里有的字段，不用回表
SELECT id, status FROM orders WHERE status = 1 ORDER BY id LIMIT 1000000, 10;
```

前提：建联合索引 `(status, id)`

```
索引里已包含 id 和 status → 不用回表 → 省掉随机 IO
但还是要扫 100 万行（减少一半时间，没根治）

适用场景：分页量不太大（几十万以内），能接受只返回部分字段
```

### 方案2：延迟关联（核心优化）

```sql
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders WHERE status = 1 ORDER BY id LIMIT 1000000, 10
) AS tmp ON o.id = tmp.id;
```

原理：先通过小索引定位 ID，再用 ID 回表取完整数据

```
直接 LIMIT：
  扫索引 → 找到 100 万行 → 每行都回表取 * → 随机 IO × 100 万

延迟关联：
  子查询只扫索引 → 找到 100 万行（不回表，只取 id）
  → 最后 10 行才回表 → 随机 IO × 10
```

```
对比：
| 方式          | 回表次数      | 扫描方式     |
|--------------|-------------|-------------|
| 直接 LIMIT    | 100 万次    | 逐行回表     |
| 延迟关联      | 10 次       | 先扫索引再回表|
```

### 方案3：游标分页（性能最好）

```sql
-- 上一页最后一条的 id 是 1000000
SELECT * FROM orders 
WHERE status = 1 AND id > 1000000 
ORDER BY id 
LIMIT 10;
```

```
普通分页：LIMIT 1000000, 10 → 扫描 100 万行然后丢弃
游标分页：WHERE id > 1000000 LIMIT 10
  → 直接定位到 id=1000000 的位置（B+ 树查找 O(logN)）
  → 往后取 10 行
  → 时间复杂度 O(logN + 10)，跟 offset 大小无关
```

适合场景：
```
✅ 无限滚动（抖音/微信朋友圈/小红书）
✅ "加载更多"（不是跳到第 N 页）
✅ 后台管理列表（只有上一页/下一页）
✅ 大数据量导出（批量处理，每次拉 1000 条）

❌ 不适合：
  需要跳到第 N 页（电商筛选页）
  需要显示"共 12345 页，第 500 页"
```

### 方案4：业务限制页数范围（根治）

```
淘宝商品搜索：最多展示 100 页（每页 20 条 = 2000 条）
→ 深分页问题根本不存在
→ 这是最实用的方案，但不是技术方案
```

## 三、Oracle vs MySQL

```
| 维度           | Oracle                          | MySQL                     |
|---------------|--------------------------------|---------------------------|
| CBO 优化器     | 非常聪明，自动选最优执行计划       | 相对笨，LIMIT 深分页直接扫   |
| 索引利用       | 反向索引、函数索引、位图索引       | 只有 B+ 树，功能少          |
| 分页语法       | ROW_NUMBER() 语义清晰           | LIMIT offset, count 模糊   |
| 内存管理       | SGA 自动管理，热数据常驻内存       | Buffer Pool 靠 OS          |
| 并行查询       | 多线程并行扫描                   | 社区版没有                  |

MySQL 没打算优化 LIMIT 深分页：
  → LIMIT offset, count 是 SQL 标准语法
  → 改了破坏兼容性
  → 根本问题是 B+ 树不能直接定位"第 N 条"
  → 正确做法是开发者用游标分页或延迟关联
```

## 四、生产环境完整方案

```sql
-- 场景1：后台管理系统，需要深分页 + 排序（能跳页）
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders WHERE status = 1 
    ORDER BY create_time DESC, id DESC
    LIMIT 1000000, 10
) tmp ON o.id = tmp.id;
-- 建索引：(status, create_time, id)

-- 场景2：移动端列表，无限滚动（不能跳页，性能最好）
SELECT * FROM orders 
WHERE status = 1 
  AND (create_time, id) < ('2026-07-01 12:00:00', 12345)
ORDER BY create_time DESC, id DESC
LIMIT 10;
-- 建索引：(status, create_time, id)
```

## 五、总结

```
| 方案       | 原理                | 性能     | 适用场景              | 限制        |
|-----------|--------------------|---------|----------------------|-----------|
| 覆盖索引   | 不回表              | 中      | 能接受部分字段          | 字段受限    |
| 延迟关联   | 先扫索引再回表        | 好      | 需要跳页的深分页        | 仍要扫描offset行|
| 游标分页   | WHERE id > 上次ID   | 最好    | 无限滚动/加载更多      | 不能跳页    |
| 缩小offset | 业务限制页数范围      | 根治    | 前端可控的分页范围      | 不是技术方案 |
```

## GPT 纠错

- GPT 纠错：覆盖索引和延迟关联减少的是回表成本，通常仍需扫描或跳过大量 offset 索引项，因此不能把它们描述成消除了深分页扫描。
- GPT 纠错：游标分页需要稳定且唯一的排序键，例如 `(created_at, id)`；只写 `id > lastId` 无法覆盖任意排序和倒序场景。
- GPT 纠错：游标分页在并发插入、删除、更新排序字段时仍要定义一致性语义，可能出现漏读或重复，不能只写“性能最好”。
- GPT 纠错：“Oracle 优化器非常聪明、MySQL 相对笨”和“MySQL 没打算优化深分页”不是严谨技术结论。两者都依赖索引、统计信息、执行计划和具体 SQL。
- GPT 纠错：限制页数只是产品层规避，并没有根治数据库层深 offset 的成本。
