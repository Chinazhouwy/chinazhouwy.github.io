## 核心概念

### 一、四种记忆类型

**1. 短期记忆（Working Memory）**
- **本质**：当前上下文窗口（context window）
- **容量**：受模型限制，比如 128K tokens
- **策略**：
  - **滑动窗口**：保留最近 N 轮对话
  - **重要性评分**：给每条消息打分，低分的先丢
  - **压缩摘要**：把旧对话压缩成一段摘要塞回去
- **什么时候丢**：上下文快满时（比如超过 80%）

**2. 长期记忆（Long-term Memory）**
- **本质**：持久化存储的历史信息
- **存储位置**：markdown 文件、SQLite、向量数据库
- **内容**：用户偏好、历史决策、关键事实
- **写入时机**：对话结束时、压缩前、用户明确要求记住时

**3. 摘要记忆（Summary Memory）**
- **本质**：对长对话的压缩版本
- **解决的问题**：上下文窗口不够用
- **实现方式**：
  ```
  原始：30轮对话（50K tokens）
      ↓ 摘要
  压缩后：一段总结（2K tokens）+ 最近5轮原文
  ```

**4. 向量记忆（Episodic Memory）**
- **本质**：用向量数据库存 + 检索
- **流程**：
  ```
  存储：文本 → Embedding → 向量数据库
  检索：查询 → Embedding → 相似度搜索 → Top-K → 塞回上下文
  ```

### 二、Memory vs RAG 的区别

| | Memory | RAG |
|---|---|---|
| **内容来源** | 对话中产生的 | 外部文档（人写的） |
| **谁产出** | 模型参与生成 | 模型没参与 |
| **例子** | "用户喜欢并发题" | 《Redis 设计与实现》文档 |
| **目的** | 保持对话连贯 | 查事实性信息 |

**一句话**：Memory = "我和你聊过什么"，RAG = "我查了什么资料"

---

### 三、短期记忆截取：主流方案

**方案一：滑动窗口（最简单）**
- 保留最近 N 轮，丢掉旧的
- **问题**：重要信息在第 10 轮，最近 3 轮是"谢谢"，直接丢了

**方案二：重要性打分（主流）**
- 给每条消息打分，低分先丢

| 打分维度 | 高分 | 低分 |
|------|------|------|
| 信息密度 | "我需要用 Redis 做分布式锁" | "好的"、"谢谢" |
| 实体关键词 | 包含类名、API、技术术语 | 纯语气词 |
| 时间衰减 | 越新越重要 | 越旧越不重要 |
| 角色 | 用户消息 > 系统消息 | tool call 的输出 |
| 是否被引用 | 后续对话提到过的内容 | 之后再也没提过的 |

**打分公式**（简化版）：
```
score = 信息密度 × 0.4 + 实体密度 × 0.3 + 时效性 × 0.2 + 角色权重 × 0.1
```

**方案三：摘要压缩（最常用）**
- 不是"丢掉"，而是"压缩"
- 把旧对话压缩成结构化摘要

**方案四：混合策略（生产推荐）**
1. 规则过滤：丢掉纯语气词（"谢谢"、"好的"、"收到"）
2. 重要性打分：给剩余消息打分
3. 分层保留：
   - 最近 3 轮：原文保留
   - 3-10 轮：高分保留原文，低分压缩成摘要
   - 10 轮以前：全部压缩成一段摘要
4. 关键信息提取：用户偏好、决策、待办单独存

### 四、话题切换检测

**场景**：
```
用户：Redis 缓存穿透怎么解决？
Agent：用布隆过滤器 + 空值缓存...
用户：今天天气怎么样？（岔出去）
Agent：今天晴，25度...
用户：回到刚才的 Redis 问题，布隆过滤器误判怎么办？（拉回来）
```

**检测信号**：
1. **语义相似度骤降**：上一条是技术问题，这一条突然是生活问题
2. **实体断裂**：上一条有 "Redis"、"缓存"，这一条变成 "天气"、"温度"
3. **意图跳转**：从 QA 意图变成闲聊意图

**处理策略**：
- **低价值闲聊**（天气、问候）→ 直接压缩成一句话
- **有价值但暂时不相关** → 存到长期记忆
- **记录回归锚点**：用户在第 N 轮讨论过 Redis，第 M 轮岔出去，第 K 轮回来了

### 五、MemGPT 完整处理链路

```
用户发消息
    ↓
1. 预处理层：意图识别、实体提取、话题分类
    ↓
2. 话题检测层：语义相似度对比，判断是否切换
    ↓
3. 短期记忆管理层：检查窗口使用率，压缩/截取/过滤
    ↓
4. 长期记忆检索层：判断上下文够不够，不够就向量检索
    ↓
5. 上下文组装层：System Prompt + 用户画像 + 摘要 + 检索记忆 + 最近原文 + 当前消息
    ↓
6. 模型推理层：调用 LLM，可能触发反思动作
    ↓
7. 后处理层：存储偏好、更新画像、记录回归锚点
    ↓
返回给用户
```

### 六、七个切面总结

| 切面 | 做什么 | 为什么需要 |
|------|--------|-----------|
| **1. 预处理** | 意图识别、实体提取 | 给后续环节提供信号 |
| **2. 话题检测** | 语义相似度对比，判断是否切换 | 决定要不要清理上下文 |
| **3. 短期记忆管理** | 压缩、截取、过滤 | 控制 token 用量 |
| **4. 长期记忆检索** | 向量搜索相关记忆 | 补全上下文缺失的信息 |
| **5. 上下文组装** | 把所有材料拼好送入模型 | 决定模型能看到什么 |
| **6. 模型推理** | 生成回答，可能触发反思 | 实际回答问题 |
| **7. 后处理** | 存储、更新画像、记录锚点 | 为下一轮做准备 |

## 追问 + 纠正记录

### 追问 1：短期记忆截取策略

**场景**：最近几轮都是"谢谢"、"好的"、"收到"，但重要的需求定义在第 10 轮之前。

**纠正**：不是简单截取，是"过滤低价值 + 打分 + 摘要压缩"的组合。主流方案是混合策略。

### 追问 2：话题切换处理

**场景**：对话中岔出去聊了无关话题，又拉回来。

**纠正**：通过语义相似度骤降检测话题切换。检测到后，低价值闲聊直接压缩，有价值但暂时不相关的存到长期记忆。同时记录回归锚点。

## 最终结论

| 用户问题 | 结论 |
|---------|------|
| 短期记忆怎么截取 | 混合策略：过滤低价值 + 重要性打分 + 分层保留 + 摘要压缩 |
| 怎么判断上下文够不够 | 语义匹配，不够就去长期记忆检索 |
| Memory vs RAG 区别 | Memory 是对话中产生的，RAG 是外部知识库 |
| 岔出去的话题怎么处理 | 检测切换 → 分类处理 → 清理上下文 → 保留回归锚点 |

---

## 这次讨论的收获

1. **短期记忆截取不是简单滑动窗口**：需要打分+压缩的组合策略
2. **话题切换检测是关键**：语义相似度骤降是主要信号
3. **MemGPT 的核心思想**：让 Agent 自己管理记忆，像操作系统管理内存一样
4. **七个切面**：预处理→话题检测→短期管理→长期检索→上下文组装→模型推理→后处理
5. **Memory vs RAG**：Memory 是对话中产生的，RAG 是外部文档
