## 一、JDK 1.7 vs 1.8 底层结构对比

| 特性 | JDK 1.7 | JDK 1.8 |
|------|---------|---------|
| 数据结构 | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| 哈希冲突 | 链表（头插法） | 链表（尾插法）→ 红黑树 |
| 插入方式 | 头插法（多线程不安全） | 尾插法（多线程仍不安全） |
| 查询复杂度 | O(n) 最坏 | O(logn) 红黑树 / O(n) 链表 |

**关键区别**：
- 1.7 链表是**头插法**（新节点插到链表头部），扩容时会反转链表顺序
- 1.8 改成**尾插法**（新节点插到链表尾部），扩容时保持顺序不变

## 二、链表转红黑树的条件

```java
// 源码：HashMap.java
final void putVal(int hash, K key, V value, ...) {
    // ...
    if (binCount >= TREEIFY_THRESHOLD - 1) // binCount >= 7，即链表长度 >= 8
        treeifyBin(tab, hash);
}

final void treeifyBin(Node<K,V>[] tab, int hash) {
    // 关键：数组长度必须 >= 64
    if (tab == null || (n = tab.length) < MIN_TREEIFY_CAPACITY)
        resize(); // 数组长度 < 64 时，优先扩容而不是转红黑树
    // ...
}
```

**两个条件必须同时满足**：
1. 链表长度 ≥ 8（TREEIFY_THRESHOLD）
2. 数组长度 ≥ 64（MIN_TREEIFY_CAPACITY）

**为什么不是 1 或 16？**
- 阈值 8 是基于**泊松分布**（Poisson Distribution）的概率计算
- 在正常哈希分布下，链表长度达到 8 的概率只有 **0.00000006**
- 也就是说，几乎所有情况下链表都不会转红黑树，红黑树是**极端情况的保底**
- 如果阈值设太小（如 1），频繁转换反而降低性能

## 三、HashMap 为什么线程不安全？

**JDK 1.7 死循环问题**（经典面试考点）：

```
线程A扩容前：  A → B → C（null）
线程B扩容前：  A → B → C（null）

线程A执行到一半（A.next = B，还没处理B）
线程B完成扩容，使用头插法，链表反转：C → B → A

线程A继续：B.next = A，A.next = null
结果：B → A → null，但 B 的 next 是 A，A 的 next 是 null
实际形成：B → A → B → A → ... 死循环！
```

**JDK 1.8 改进**：
- 改用尾插法，扩容时保持链表顺序，**不会形成环**
- 但仍不是线程安全的：多线程 put 会**丢失数据**（覆盖写）

## 四、负载因子为什么是 0.75？

```java
static final float DEFAULT_LOAD_FACTOR = 0.75f;
```

**设计权衡**：
- **太小（如 0.5）**：空间浪费，扩容频繁
- **太大（如 1.0）**：哈希冲突多，查询变慢
- **0.75 是折中**：空间和时间的平衡点

**更深层原因**：
- 0.75 接近**泊松分布**的期望值
- 配合哈希扰动函数，能让桶中元素数量服从泊松分布
- 长度为 8 的概率极低（0.00000006），所以红黑树转换是极端保底

## 五、面试回答模板

> **JDK 1.7 是数组+链表，1.8 改成数组+链表+红黑树。**
> 
> **链表转红黑树有两个条件**：链表长度≥8 且数组长度≥64。阈值 8 是基于泊松分布计算的，正常情况下几乎不会触发，是极端情况的保底。
> 
> **HashMap 线程不安全**：1.7 多线程扩容时，头插法会导致链表成环死循环；1.8 改成尾插法解决了环问题，但仍会丢失数据。
> 
> **负载因子 0.75** 是空间和时间的折中，配合哈希扰动函数让桶内元素服从泊松分布。

## 六、用户追问 + 纠正记录

1. **头插法 vs 尾插法扩容时顺序变化** → 用图示详细解释，头插法扩容时链表顺序会反转，尾插法保持不变
2. **Map 无所谓顺序** → 用户指出 Map 本身无序，顺序反转不是问题，真正问题是多线程下头插法会导致链表成环死循环
3. **多线程就不该用 Map** → 用户指出正确用法：多线程应该用 ConcurrentHashMap，HashMap 线程安全问题是历史遗留
4. **泊松分布是什么** → 用奶茶店例子解释，描述稀有事件发生次数的概率分布，HashMap 中链表长度达到 8 的概率只有千万分之六

## 七、最终结论

### 骨架答案（一句话核心）
> HashMap 1.7 是数组+链表（头插法），1.8 增加红黑树（尾插法）。链表转红黑树需要链表长度≥8 且数组长度≥64，阈值 8 基于泊松分布。线程安全问题的本质是多线程不该用 HashMap，应该用 ConcurrentHashMap。

## 八、这次讨论的收获

1. **链表转红黑树的两个条件**：不只是链表长度，还需要数组长度≥64
2. **头插法的真正问题**：不是顺序反转（Map 无序），而是多线程下会形成环导致死循环
3. **泊松分布的实际意义**：解释了为什么阈值设为 8，极端情况的保底机制
4. **正确用法**：多线程环境应该用 ConcurrentHashMap，而不是纠结 HashMap 的线程安全问题
