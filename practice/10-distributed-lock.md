### 追问+纠正记录
1. **看门狗是 Redisson 的功能**，不是 Redis 本身的；Redis 只有 key 过期时间
2. **ZK 是临时顺序节点 + Watcher**，不是"某个节点"；临时=客户端断开自动删除，顺序=解决惊群效应
3. **Redis 锁核心问题**：锁过期但业务没完、主从切换丢锁、时钟漂移
4. **ZK 加锁流程**：创建临时顺序节点 → 判断是否最小 → 不是则 Watcher 监听前一个节点
5. **DB 两种方案**：唯一索引 INSERT 冲突检测 / SELECT FOR UPDATE 行锁
6. **Redisson 看门狗原理**：加锁后每 10 秒自动续期（默认 30 秒），unlock 停止

### 最终结论
Redis=AP 高性能但主从可能丢锁；ZK=CP 强一致但性能差；DB 最简单但性能最差。选型看场景。

### SETNX+过期时间的三大问题

```java
// 常见错误写法
redis.setnx("lock", "1");
redis.expire("lock", 10);  // 两行不是原子操作！
```

**问题1：非原子操作**
```
线程A: SETNX成功 → 还没设置过期时间 → 宕机了
结果：锁永远不过期 → 死锁
```

**问题2：锁过期但业务没完成**
```
线程A: 获取锁，过期时间10秒
线程A: 业务执行了15秒（比如慢SQL）
第10秒：锁自动过期
线程B: 获取到锁 → 两个线程同时操作 → 数据不一致！
```

**问题3：锁被别人释放**
```
线程A: 获取锁
线程B: 等待超时，强制释放（DEL lock）→ 释放了A的锁！
线程A: 业务还没做完，锁没了
```

### Redisson看门狗机制

```java
RLock lock = redisson.getLock("myLock");
lock.lock();  // 默认30秒过期，看门狗自动续期

try {
    // 业务逻辑
    // 看门狗每10秒检查一次，如果业务还在跑，自动续期到30秒
} finally {
    lock.unlock();
}
```

**看门狗工作原理：**
```
获取锁（30秒过期） → 启动看门狗后台线程
    ↓
每 30/3 = 10秒 检查一次
    ├── 业务还在跑？ → 续期到30秒
    ├── 业务完成了？ → 停止看门狗
    └── 客户端宕机？ → 看门狗也死了 → 锁自然过期释放
```

**为什么是30秒？** 太短网络抖动可能来不及续期，太长宕机后锁释放太慢。30秒是经验值。

### RedLock算法

**主从模式的问题：**
```
客户端A在Master上获取锁
Master还没同步给Slave就宕机了
Slave升级为Master
客户端B在新Master上获取到同一把锁 → 两个客户端同时持有锁！
```

**RedLock（Redis作者Antirez提出）：**
```
1. 获取当前时间 T1
2. 依次向N个独立Redis实例请求加锁（N通常为5个）
3. 如果在超过半数（≥3）的实例上加锁成功
   且总耗时 < 锁的过期时间 → 加锁成功
4. 锁的有效时间 = 过期时间 - (T2 - T1)
5. 如果加锁失败，向所有实例发送释放锁请求
```

**争议：** Martin Kleppmann（《DDIA》作者）质疑RedLock，认为时钟漂移和GC暂停可能导致不安全。

### 追问：GC暂停和时钟漂移

**GC暂停：** Java的GC会让整个JVM暂停（Stop-The-World），如果暂停时间超过锁的过期时间，锁过期了但客户端还以为自己持有锁。

```
客户端A获取锁 → GC暂停35秒 → 锁30秒过期 → 客户端B获取到锁
→ A醒来继续执行 → 两个客户端同时操作
```

**时钟漂移：** 每台机器的时钟不完全同步，RedLock依赖多节点时钟判断锁状态，时钟不一致会导致误判。

**关键结论：** 单节点Redisson+看门狗，GC暂停时看门狗也暂停，同生共死，相对安全。RedLock依赖远程节点时钟，风险更大。

### 追问：Fencing Token

给每次锁分配一个递增编号，写操作时带上编号，编号不对就拒绝。

```
客户端A获取锁，token=33 → 写数据库，token=33
客户端B获取锁，token=34 → 写数据库，token=34
客户端A从GC恢复，带着token=33写数据库
→ 数据库检查：33 < 34 → 拒绝！
```

### 追问：GC暂停是所有分布式锁的通病

**用户原话：** "单节点Redis也存在GC暂停，比如A机器拿到锁，GC暂停，时间到了，还在GC，B机器拿到锁了"

**回答：** 完全正确。分布式锁只能防止"正常情况"下的并发。GC暂停超时是所有方案的共同弱点，真正的安全保障靠业务层面兜底（数据库约束、版本号、幂等设计）。

### 生产事故故事：超卖问题排查

**背景：** 电商系统限量抢购，Redisson分布式锁，偶尔库存超卖。

**排查过程：**
1. Review代码，锁加了没问题
2. 写脚本提取超卖订单时间点，和GC日志交叉对比
3. 发现每次超卖前30-40秒都有一次Full GC（35秒）
4. 原因：热点商品缓存用HashMap存了10万+数据，没设过期，老年代满触发Full GC

**根因：** Full GC导致STW 35秒 > 锁过期30秒 → 锁失效 → 双线程同时扣库存 → 超卖

**解决方案：**
1. 紧急：热点缓存换Caffeine，设过期+容量上限
2. 中期：数据库乐观锁兜底 `UPDATE SET count=count-1 WHERE id=? AND count>0 AND version=?`
3. 长期：库存监控告警 + SkyWalking链路追踪

### 一句话总结

> **SETNX 命令本身是原子的，真正的问题是 SETNX + EXPIRE 分两条命令不是原子操作，可能导致锁没有过期时间；生产用 `SET key value NX PX/EX` 或 Lua。Redisson 看门狗能自动续期，但 GC/STW 超过续期间隔仍可能导致锁过期；集群下 RedLock 用多独立节点降低主从切换丢锁风险但有争议，最终仍要靠业务幂等、乐观锁或 fencing token 兜底。**
