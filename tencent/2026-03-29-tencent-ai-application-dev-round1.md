## Q1: 项目里是把 skill 直接塞进 system prompt 的，如果 skill 太多，占用上下文窗口太大，怎么处理

### 答题思路

这是一道**工程落地题**，面试官想看你在实际项目中如何解决token膨胀问题。回答要抓住三个关键词：**外部注册表、按需注入、skill routing**。

### 深度解答

不能把所有 skill 常驻塞进 system prompt，这样会带来三个问题：
1. **上下文窗口被占满**
2. **候选技能噪声太大**
3. **模型在选择 skill 时更容易混淆**

更合理的方式是把 skill 做成**外部注册表**，system prompt 里只保留最小规则和调用协议，真正的 skill 描述按需动态注入。

**常见做法**：先做一层 **skill routing**。可以用规则、分类模型或者向量检索先筛出 topk skill，再把这几个 skill 的 description、参数 schema 和 few-shot 示例拼进 prompt。如果 skill 本身很长，还要把 skill 信息拆成**摘要版和完整版**，先注入摘要，模型确定要调用后再加载详细说明。这样可以显著压缩 token。

## Q2: 如果两个 skill 实际内容不一样，但是 description 很相似，导致 agent 每次加载错 skill，怎么解决

### 答题思路

这道题考察**工程判别能力**，面试官想知道你在skill消歧上的实践。核心思路是**给skill增加更强的判别信息**。

### 深度解答

description 相似时，单靠自然语言匹配很容易误召回。解决思路是给 skill 增加更强的判别信息，而不是继续堆模糊描述。

**四步解决方案：**

1. **结构化定义**：把 skill 从纯文本 description 升级成结构化定义，包括适用场景、禁用场景、输入参数、返回格式、前置条件、失败条件和调用示例。

2. **两阶段选择**：先召回 topk，再让模型根据参数约束和样例做 rerank。

3. **负样本示例**：加入负样本示例，明确告诉模型什么问题不应该调用这个 skill。

4. **Schema校验**：执行前做 schema 校验和 rule check，即使模型选错，也不能继续误调。

**补充**：如果两个 skill 语义长期接近，通常还要考虑在系统设计层面**合并成一个 skill**，再由内部参数路由，不然前面再怎么写 prompt，后面也会持续混。

## Q3: Agent 设计里你觉得最重要的部分是什么

### 答题思路

这是一道**开放设计题**，面试官想看你的架构理解深度。核心观点：**上下文工程 > 模型能力**。

### 深度解答

最重要的是**上下文工程**。Agent 最终不是单轮问答，而是一个持续决策系统。模型能看到什么信息、这些信息的顺序和优先级是什么、哪些内容是规则、哪些内容是证据、哪些内容只是历史状态，这些都会直接影响 Agent 的稳定性。

**上下文工程做得好**，模型会更像一个受控执行器，知道当前目标、约束条件、可调用工具、历史状态和输出格式。

**上下文工程做得差**，模型即使能力强，也会出现目标漂移、工具误选、记忆污染、长对话失真和推理跑偏。

所以 Agent 设计的核心不是"给模型更多信息"，而是"**给模型正确的信息**"。

## Q4: 上下文工程有哪些要注意的点

### 答题思路

紧接Q3，考察你对上下文工程的**工程细节**掌握。要从角色隔离、信息裁剪、格式控制、来源可信度四个维度展开。

### 深度解答

**1. 角色隔离**
system、developer、user、tool output、memory 的优先级和边界要清楚，不能把外部文本和系统规则混在一起。

**2. 信息裁剪**
只保留当前任务相关的内容，避免把所有历史对话、所有检索结果和所有工具说明一股脑灌进去。

**3. 格式控制**
结构化上下文通常比纯自然语言更稳，比如工具结果尽量用 JSON、任务状态单独字段化、长历史做摘要而不是堆原文。

**4. 来源可信度分层**
上下文中的来源可信度必须分层，系统规则优先于用户输入，工具结果优先于模型猜测，知识证据优先于无依据生成。

**5. 长任务处理**
如果是长任务，还要做上下文压缩和过期淘汰，不然越到后面越容易发生状态漂移和旧信息污染新决策。

## Q5: 你怎么设计 Agent 的长期记忆写回策略、衰减策略和冲突消解

### 答题思路

考察**记忆系统设计**能力。长期记忆不能等价于"把所有历史都存起来"，要回答三个问题：什么值得写回、什么时候应该忘、冲突信息怎么处理。

### 深度解答

**写回策略**
通常只保存高价值稳定信息，比如用户偏好、身份属性、长期任务目标、反复出现的业务事实，而不是每一句对话都入库。写回时最好做**结构化抽取**，比如把"用户更喜欢用 Python"写成 profile 字段，而不是原文整段存储。

**衰减策略**
记忆需要有时效性和权重，短期热点信息可以高权重，但随着时间衰减；长期稳定偏好可以低频刷新但不轻易删除。

**冲突消解**
一般要结合时间戳、来源可信度和置信度，新的高可信事实覆盖旧事实，低可信内容只作为候选，不直接替换。

**进阶设计**：如果更严格一些，可以把长期记忆设计成**事件流 + 物化视图**。也就是所有记忆写入先存成 event，再异步聚合成 profile。这样既方便审计，也方便回滚。

## Q6: 你怎么做并行 Tool Calling，既提升吞吐又保证一致性和可回放

### 答题思路

考察**工程架构能力**。并行 Tool Calling 的核心不是"同时调多个接口"，而是要解决依赖关系、状态一致性、结果合并和失败回滚。

### 深度解答

**依赖分析**
只有互不依赖的工具才能并行，比如天气查询和汇率查询可以一起发；如果后一个工具依赖前一个工具的结果，就必须串行。

**一致性保证**
一般不会让多个工具直接修改共享状态，而是采用"**工具执行结果先写到临时观察区，再由调度器统一合并**"的方式。这样可以避免并发写导致状态污染。

**可回放**
每次调用都带 trace_id、step_id、输入参数、输出结果和耗时，最后组成完整执行 DAG。失败处理上可以按工具粒度重试，不能因为一个工具超时就把全部结果丢掉。

**副作用处理**
如果涉及副作用操作，比如发消息、创建订单、提工单，并行执行前通常还要加**幂等键**，不然重复重试会造成真实业务重复写入。

## Q7: 了解主流的 Agent 设计吗

### 答题思路

考察**架构视野**。主流 Agent 设计一般分成几类，要能说清楚每类的特点和适用场景。

### 深度解答

**1. ReAct（最经典）**
边推理边行动，模型通过思考、调用工具、观察结果、继续思考来完成任务。

**2. Plan-and-Execute**
先做规划，再按步骤执行，比纯 ReAct 更稳，适合长任务。

**3. Router 模式**
先做任务分类，再路由给不同的子 Agent 或工具链。

**4. Workflow + Agent**
固定流程里嵌入局部自主决策，这种方式在线上业务里用得很多。

**多 Agent 场景**：
- **Supervisor 模式**：由主 Agent 拆解任务并分派给多个子 Agent
- **Blackboard 模式**：多个 Agent 通过共享状态协作
- 纯点对点自由协作也有，但线上比较少，因为不容易治理

## Q8: MCP、A2A、普通 Function Calling 三者有什么本质区别，工程里怎么选

### 答题思路

考察**协议理解深度**。三者解决的问题层次不同，要从定义、适用场景、工程选型三个角度对比。

### 深度解答

**Function Calling**
模型和宿主应用之间的本地调用协议。模型输出结构化的函数名 + 参数，宿主执行后把结果喂回模型。适合**单应用内**的工具调用，比如用户问天气，模型调一个 get_weather 函数。

**MCP（Model Context Protocol）**
把工具能力从应用里抽出来，变成**标准化的服务接口**。任何支持 MCP 的客户端都能调用，解决了工具生态碎片化问题。适合**跨应用、跨平台**的工具共享。

**A2A（Agent-to-Agent）**
解决的是**Agent 之间互相发现和协作**的问题。一个 Agent 可以通过 A2A 协议找到另一个 Agent，发起任务、传递上下文、获取结果。适合**多 Agent 协作**场景。

**工程选型原则**：
- 单应用内工具调用 → Function Calling
- 需要共享工具生态 → MCP
- 多 Agent 协作 → A2A
- 实际项目中三者常常混合使用
