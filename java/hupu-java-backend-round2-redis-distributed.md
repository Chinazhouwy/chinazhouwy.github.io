# 虎扑后端Java二面面经 — Redis+分布式

> 来源：小红书 | 时长：45分钟 | 侧重：Redis + 分布式核心（中间件+分布式场景）

## 面试题

### 一、Redis 核心原理

**1. Redis 为什么性能极高？**
- 纯内存操作，磁盘IO极少
- 单线程模型，无锁竞争
- 数据结构高度优化
- 多路IO复用模型
- 简单协议，解析开销小

**2. RDB 和 AOF 持久化区别？线上如何选型？**

| 维度 | RDB | AOF |
|------|-----|-----|
| 方式 | 快照 | 追加日志 |
| 文件大小 | 小 | 大 |
| 恢复速度 | 快 | 慢 |
| 数据安全 | 可能丢失数据 | 丢失少（取决于刷盘策略） |

**混合持久化**：Redis 4.0+，RDB 做全量快照 + AOF 增量日志，兼顾恢复速度和数据安全。

### 二、分布式实战

**1. Redis 分布式锁原理？如何解决超时/误删/续期？**
- `SET NX EX` 实现加锁，唯一 ID 标识锁持有者
- 超时：**看门狗续期机制**（Redisson WatchDog）
- 误删：校验唯一 ID 再删除
- 死锁：设置过期时间

**2. 缓存穿透、击穿、雪崩的区别和解决方案？**

| 问题 | 现象 | 解决方案 |
|------|------|----------|
| 穿透 | 查询不存在数据 | 布隆过滤器、空值缓存、接口校验 |
| 击穿 | 热点 key 过期，大量请求打 DB | 互斥锁、热点 key 永不过期 |
| 雪崩 | 大量 key 同时过期/缓存宕机 | 过期时间随机、集群高可用、多级缓存、限流降级 |

**3. CAP 和 BASE 理论？分布式场景如何取舍？**
- **CAP**：一致性(C)、可用性(A)、分区容错性(P)，分布式必须满足P，C和A只能二选一
- **BASE**：基本可用(BA)、软状态(S)、最终一致性(E)
- 互联网高并发业务优先保证可用性，采用**最终一致性**
