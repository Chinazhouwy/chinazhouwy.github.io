### 2. 业务情景核心题

#### (1) 顺丰双十一/618 包裹量暴增，订单、物流轨迹系统压力剧增，你作为后端开发如何做系统扩容与流量保障？

**回答框架（分层扩容策略）：**

**① 流量入口层：**
- **负载均衡扩容**：提前评估峰值 QPS，LVS/Nginx 水平扩展，增加 upstream 节点
- **限流降级**：接入层 Sentinel/Guava RateLimiter 做排队/限流，非核心功能降级
- **流量调度**：按区域/仓库维度做流量打散，避免单点热点

**② 缓存层：**
- **多级缓存**：本地缓存（Caffeine）+ 分布式缓存（Redis Cluster）
- **热点数据预热**：大促前将热门线路/区域的物流状态预加载到缓存
- **缓存策略**：轨迹快照缓存（最近 3 天），历史轨迹走 ES

**③ 应用层：**
- **弹性伸缩**：K8s HPA 基于 QPS/CPU 自动扩缩 Pod，大促前提前扩容 2-3 倍
- **异步化**：订单创建/轨迹更新写入 MQ 异步处理，削峰填谷
- **读写分离**：订单查询走只读节点，更新走主库

**④ 数据层：**
- **分库分表**：按订单 ID hash / 时间 / 区域分片
- **读写分离**：主库写入 + 从库查询（配合延迟容忍策略）
- **连接池调整**：大促前调大 maxActive，预留连接余量

**⑤ 兜底策略：**
- **熔断**：Sentinel 熔断降级，非关键服务（如天气预报、广告推荐）直接降级
- **兜底页**：极端情况返回"查询繁忙，稍后再试"，保证核心下单/轨迹链路
- **提前压测**：全链路压测 + 容量评估，找瓶颈

#### (2) 高峰期快递小哥运力不足、订单积压，从技术层面如何优化订单分配、揽收调度系统？

**回答框架：**

**① 智能调度算法：**
- **运力预测**：基于历史数据 + 天气/节假日特征，预估每个区域的实时运力缺口
- **动态分配**：
  - 基础：按区域+距离分配（就近原则）
  - 进阶：带权分配（快递员评分×距离×负载因子）
  - 高阶：约束优化（时间窗约束，类似 Vehicle Routing Problem）
- **抢单 vs 派单混合**：核心区域自动派单，边缘区域抢单+激励

**② 弹性运力池：**
- **临时运力接入**：系统预留临时快递员接入通道，高峰期启动众包/兼职运力
- **运力分级**：全职（必达）→ 兼职（弹性）→ 众包（兜底）
- **压力预警**：运力利用率 > 80% 自动预警，触发扩招/调度策略切换

**③ 系统架构支撑：**
- **异步调度**：订单积压后，调度任务写入延迟队列，逐批释放避免瞬间压力
- **实时推送**：WebSocket / 长轮询 实时推送订单到快递员 App
- **兜底重试**：分配失败的订单进入重试队列，等待运力释放

**工程踩坑：**
- 分配算法不能用全局最优（NP-hard），用贪心+局部优化，实时计算（毫秒级）
- 快递员 App 网络不稳定，需要消息可靠到达 + 离线缓存
- 订单超时未揽收需自动重新分配，避免"死单"

#### (3) 物流轨迹数据实时更新，如何保证用户查询实时性 + 系统高可用？

**回答框架：**

**架构策略：** 写入异步化 + 查询缓存化 + 存储分层化

**写入链路：**
```
快递员/分拣设备 → MQ（Kafka/RocketMQ）→ 轨迹服务 → 
   ├─ 实时写入 Redis（最新轨迹，TTL 7天）
   └─ 批量写入 ES（全量轨迹，长期存储）
```

**查询链路：**
```
用户查询 → Redis 查最新轨迹（99%命中）→ 未命中查 ES → 返回
```

**关键设计：**

| 环节 | 方案 | 说明 |
|------|------|------|
| 写入 | 异步 MQ | 轨迹写入抗百万级 TPS，MQ 削峰填谷 |
| 最新状态 | Redis String/Hash | 订单号→最新轨迹，TTL 7天，单次查询 < 1ms |
| 轨迹列表 | Redis List / SortedSet | 按时间倒序，最近 20 条惰性加载 |
| 历史轨迹 | ES | 按订单号 + 时间索引，冷热分层 |
| 实时推送 | WebSocket + 心跳 | 轨迹更新时推送用户端 |

**高可用保障：**
- Redis Cluster 主从切换 + Sentinel
- ES 集群跨机房部署，分片副本 2+
- 兜底：Redis 宕机时降级查 ES（延迟增加，但可用）
- 监控：轨迹写入延迟、查询 P99、MQ 积压量

#### (4) 如果出现订单状态不一致、轨迹数据丢失问题，如何兜底修复？

**回答框架：**

**① 兜底方案（四层修复）：**

| 层面 | 方案 | 具体实现 |
|------|------|----------|
| 主动检测 | 定时对账 | 每 5 分钟对比 MQ 生产量 vs 消费量 vs 存储量 |
| 被动修复 | 用户反馈触发 | 用户上报异常→自动拉起全链路溯源任务 |
| 数据补偿 | 反向修复 | 以最新状态为准，回放 MQ 消息修复不一致数据 |
| 人工介入 | 降级到 DBA/运维 | 极端情况手动修复 + 复盘 |

**② 如何发现不一致：**
- **状态机校验**：订单状态变迁必须合法（已揽收→运输中→派送中→已签收），非法跳转报警
- **轨迹连续性校验**：轨迹时间戳不能断层，缺失超过阈值触发修复
- **对账 Job**：每天凌晨 T+1 对账，Redis vs ES vs DB 三方交叉比对

**③ 修复流程：**
```
发现不一致 → 定位故障点（写丢失/消费丢失/存储丢失）→ 
回放 MQ（Kafka 支持回溯）→ 重新写入 → 校验一致性 → 记录审计日志
```

**工程踩坑：**
- 回放时要加幂等性（避免修复过程中重复写入导致数据错乱）
- 不要"全量回放"，要"增量修复"，定位具体丢失的订单范围
- 修复 Job 要对下游做限速保护，避免回放流量打垮存储层

### 3. 技术结合业务提问

#### (1) 顺丰订单查询高频、更新频繁，数据库和缓存如何做分层设计？

**分层设计：**

```
┌─────────────────────┐
│    CDN / DNS        │ ← 静态资源加速（App 版本包等）
├─────────────────────┤
│   浏览器缓存          │ ← 短时间不变化的页面片段
├─────────────────────┤
│   Nginx Local Cache │ ← 热点数据（热门线路状态）
├─────────────────────┤
│   Redis Cluster     │ ← 订单最新状态、轨迹快照
│   - Hot Key 拆分    │
│   - 读写分离        │
├─────────────────────┤
│   Local Cache       │ ← Caffeine 缓存热点订单（毫秒级）
│   (Caffeine/Guava)  │
├─────────────────────┤
│   Elasticsearch     │ ← 轨迹历史、订单搜索
├─────────────────────┤
│   MySQL (分库分表)   │ ← 订单核心数据（最终一致性）
└─────────────────────┘
```

**3 种查询模式适配：**

| 查询类型 | 缓存策略 | 说明 |
|----------|----------|------|
| 订单最新状态 | Redis String，TTL 7d | 99% 场景，<1ms 返回 |
| 订单历史轨迹 | ES 按订单号索引 | 低频查询，允许秒级延迟 |
| 订单列表/搜索 | ES + Redis 兜底 | 分页查询，配合 MySQL 最终一致性 |

**防缓存击穿/穿透/雪崩：**
- **击穿**：互斥锁（SETNX）或本地缓存兜底
- **穿透**：布隆过滤器拦截不存在的订单号
- **雪崩**：缓存 TTL 加随机偏移，避免同时过期

#### (2) 物流海量轨迹数据存储，为什么适合用 ES，不适合纯 MySQL 存储？

| 维度 | MySQL | ES（Elasticsearch） |
|------|-------|-------------------|
| 写入吞吐 | 单库万级 TPS，分库分表后十万级 | 十万~百万级 TPS（批量写入） |
| 查询模式 | 按主键/索引查询快，模糊/聚合慢 | 全文搜索、聚合分析、范围查询天然优势 |
| 轨迹特点 | 写多读少，数据量大（日均亿级） | 写入优化 + 近实时搜索完美匹配 |
| 存储成本 | 行式存储，压缩率低 | 倒排索引+列存，压缩率高 |
| 时序特性 | 无内置时序优化 | 基于时间索引，冷热分层 |
| 扩展性 | 分库分表运维复杂 | 天然分布式，水平扩展简单 |

**轨迹数据为什么用 ES 更好：**
1. **写入模式匹配**：轨迹数据 append-only，几乎不修改，ES 的批量写入下盘效率极高
2. **查询场景丰富**：用户查"某订单轨迹"、运营查"某区域今日所有异常轨迹"、分析查"某线路平均运输时长"
3. **聚合能力强**：按省份/时间/状态聚合分析，MySQL group by 大表直接崩
4. **TTL 轻松**：ES ILM（Index Lifecycle Management）自动冷热迁移+过期删除

**混合架构建议：**
- 3 天内热数据：Redis + ES（hot tier SSD）
- 3-30 天温数据：ES（warm tier HDD）
- 30 天+冷数据：ES frozen / HDFS / 归档

#### (3) Spring Boot 自动装配原理，顺丰多个业务系统如何通过自动装配统一技术规范？

**自动装配原理：**
```
@SpringBootApplication
  └─ @EnableAutoConfiguration
       └─ 加载 META-INF/spring.factories
            └─ 按 @ConditionalOnXxx 条件匹配
                 └─ 创建 Bean（按需加载）
```

**核心机制：**
- `@ConditionalOnClass`：classpath 有某个类才加载（如 RedisTemplate 存在才配置 Redis）
- `@ConditionalOnMissingBean`：用户没自定义才自动配置
- `@ConditionalOnProperty`：配置项控制开关
- `AutoConfiguration.imports`（Spring Boot 2.7+）：替代 spring.factories

**顺丰多系统统一技术规范的方案（应用）：**
```
sf-common-parent（BOM）
  ├── sf-mybatis-starter        # 统一 MyBatis 配置（分页/加密/审计字段自动填充）
  ├── sf-redis-starter          # 统一 Redis 序列化（Protostuff/JSON）、Key 前缀规范
  ├── sf-mq-starter             # 统一 MQ 配置（序列化/重试/死信）
  ├── sf-log-starter            # 统一日志链路（TraceId 自动注入）
  ├── sf-rocketmq-starter       # RocketMQ 统一配置
  ├── sf-trace-starter          # 轨迹数据统一 SDK
  └── sf-security-starter       # 统一认证鉴权
```

**好处：**
- 新人接入成本低，一个依赖搞定全部规范
- 配置中心（Nacos/Apollo）统一管控，业务线无需关心底层
- 框架升级只需改 parent BOM 版本，全业务线同步
- 规范强制统一（如日志格式、Metrics 命名、异常处理基类）

### 4. 压力场景题

**线上突发流量峰值，系统即将熔断，你优先做哪些操作止损？事后如何复盘优化？**

**止损三步（即时操作）：**

| 优先级 | 操作 | 说明 |
|--------|------|------|
| P0 | 降级非核心功能 | 关闭推荐、报表、通知等非核心链路 |
| P0 | 扩容关键节点 | K8s 快速扩容订单/轨迹服务实例 |
| P1 | 开启限流 | 接入层 + 应用层双重限流，保核心写入 |
| P1 | 切换读写分离 | 查询全部走从库，主库专心处理写入 |
| P2 | 静态化 | 常见查询结果 CDN/本地缓存，减少后端压力 |

**事后复盘流程：**

```
1. 定位根因：是流量突增？代码bug？配置不足？
2. 全链路压测：复现流量峰值，找瓶颈点
3. 优化清单：
   ├─ 代码优化（慢SQL/GC/锁优化）
   ├─ 架构优化（异步化/缓存分层/限流阈值调整）
   ├─ 容量规划（冗余度 N+1，预留 30% buffer）
   └─ 监控增强（关键指标 P99/P999 告警）
4. 演练+预案：定期全链路压测 + 应急预案演练
```

### 5. 项目深挖

**你的过往项目，如何应对高并发、高流量场景？和顺丰物流业务的异同点？**

> 回答要点：不要生硬套用，要对比分析，体现业务理解

**相似点：**
- 核心链路必须高可用（下单=下单物流，订单状态更新=轨迹更新）
- 写多读多场景，需要缓存分层
- 异步化是标配，MQ 削峰填谷

**差异点：**
- 顺丰有强**实时性**要求（用户实时看快递位置），普通业务可能接受秒级延迟
- 顺丰有**地理位置/时空维度**，业务系统通常没有
- 顺丰需要**离线调度算法**（运力分配），普通业务以在线查询为主

**踩坑经验分享（加分）：**
- "在我的项目中，订单写入峰值 2 万 TPS，我们的方案是 XX；到了顺丰的场景（目标 10 万+），我觉得需要 XX 改进……" — 体现思考迁移能力
