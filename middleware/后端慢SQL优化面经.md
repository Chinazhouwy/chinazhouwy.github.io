**Q3: 你是怎么做这个慢 SQL 分析以及去优化的？**

**答题思路**:

慢SQL优化遵循"发现问题 → 定位瓶颈 → 分析根因 → 优化验证"的标准化流程。

**1. 发现问题**

- 开启 MySQL 慢查询日志，设置 `long_query_time = 1` 秒
- 利用 `pt-query-digest` 工具定时分析慢查询日志，按耗时/频率排序
- APM 工具（SkyWalking / ARMS）链路追踪，识别数据库层耗时占比

**2. 定位瓶颈**

- `EXPLAIN` 分析执行计划：关注 `type`（ALL 全表扫描需优化）、`key`（是否命中索引）、`rows`（扫描行数）、`Extra`（Using filesort / Using temporary 需优化）
- `SHOW PROFILE` / `SHOW PROCESSLIST` 查看实时 SQL 执行状态
- 通过 `information_schema` 查询索引使用率和冗余索引

**3. 常见优化手段**

- **索引优化**：联合索引最左匹配原则、覆盖索引避免回表、冗余索引清理
- **SQL 重写**：避免 `SELECT *`、避免隐式类型转换、用 `EXISTS` 替代 `IN`（子查询结果大时）、分页优化（延迟关联替代 `OFFSET`）
- **表结构优化**：大字段拆分、冷热数据分离、分区表
- **数据归档**：历史数据定期归档至归档表或 TiDB/ES

**4. 工程踩坑**

> ⚠️ 索引不是越多越好，写频繁的表索引过多会导致写入变慢、binlog 暴增。慢 SQL 优化一定要结合业务场景，有时业务逻辑本身不合理（如 N+1 循环查库）才是根源，光优化 SQL 没意义。

**Q4: 你这有用到 Excel 导入，你这个批量导入的上限有控制吗？**

**答题思路**:

批量导入需要考虑多个维度的上限控制：

1. **单批次大小控制**：单批次 500-1000 条为宜，避免单次插入数据量过大导致事务日志膨胀或锁竞争
2. **前端限制**：前端限制单次上传文件大小（如 10MB）、行数限制（如 1 万行）
3. **后端分批处理**：使用 `分批提交（batch commit）`，每批处理完提交一次事务，避免大事务
4. **限流保护**：结合 Guava RateLimiter 或 Sentinel，控制导入 QPS，防止打爆数据库
5. **异步处理**：超过一定量级（如 5000 行）走 MQ 异步处理，前端轮询进度

> ⚠️ 大事务隐患：单批次过大 → undo log 暴增 → 主从延迟 → 锁竞争加剧。分批提交时注意每批之间留短暂间隔（`Thread.sleep(10)`），给数据库喘息机会。

**Q5: 如果其中一批插入失败了怎么办？**

**答题思路**:

1. **事务补偿**：采用"全部或零"模式（全量成功才算成功），失败则整体回滚；或采用"逐批确认"模式（记录已成功批次 ID），只重试失败批次
2. **错误记录表**：每条数据预生成 `batch_id` + `seq`，失败时记录到 `import_error_log` 表，记录原因和原始数据
3. **幂等设计**：通过唯一键（如业务单号+批次号）保证重新导入不会重复
4. **补偿机制**：导入结束后扫描对比源文件和目标表记录数，确认一致
5. **用户反馈**：展示成功条数和失败条数，提供"下载失败数据"功能让用户修正后重试

> ⚠️ 批次粒度的选择很关键：如果一批太大，失败成本高；如果一批太小，导入太慢。建议根据线上压测确定最佳 batch size。
