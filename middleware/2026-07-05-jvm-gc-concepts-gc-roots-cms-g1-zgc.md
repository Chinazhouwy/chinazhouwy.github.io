# JVM 垃圾回收核心概念

> 来源：面试辅导对话整理 · 适用于：Java 高级面试 GC 原理

---

## 一、GC Roots 是什么

**GC Roots 是可达性分析的起点集合，不是一种对象。**

从 GC Roots 出发，沿引用链可达的对象标记为存活，不可达的就是垃圾。

### 哪些东西可以当 GC Roots

| 类型 | 说明 |
|------|------|
| **虚拟机栈（线程栈）中的局部变量** | 每个线程方法栈帧里的参数、局部变量、操作数栈引用的对象 |
| **方法区中的静态变量** | `static Object ref`，类加载后一直存在 |
| **方法区中的常量** | `static final Object`，如字符串常量池中的引用 |
| **JNI 引用** | Native 方法持有的 Java 对象引用 |
| **监视器锁持有的对象** | `synchronized(obj)` 锁住的那个对象 |
| **被 JVM 持有的对象** | 系统类加载器、异常对象、类的 Class 对象本身 |

### 关键理解

GC Roots 不是"GC 根节点"的缩写，而是"可达性分析的起点集合"。G1/ZGC/CMS 都用这个算法，只是标记阶段的实现方式不同。

## 二、并发标记（Concurrent Mark）是什么

**并发标记就是标记"谁还活着"。** 从 GC Roots 出发，沿着引用链走，把所有能走到的对象打上"存活"标记。

### 标记过程

```
GC Roots（局部变量、静态变量、常量等）
    │
    ▼
  标记 A 存活 ✓
    │
    ▼
  A.field → B，标记 B 存活 ✓
    │
    ▼
  B.field → C，标记 C 存活 ✓
    │
    ▼
  C.field = null，没引用了，停
```

### 标记存在哪里

每个对象头里有标记位（1 bit），标记时置为 1，未标记的就是 0。

```
对象头结构（简化）：
┌──────────┬──────────┬──────────┐
│ Mark Bit │ Age      │ 其他... │
└──────────┴──────────┴──────────┘
    0=垃圾    分代年龄
    1=存活
```

## 三、并发标记为什么会产生浮动垃圾（Floating Garbage）

### 根本原因

**标记阶段和用户线程并发执行，标记完之后新产生的垃圾没人再标记它了。**

### 用简单场景理解

假设整个堆里只有 3 个对象：

```
GC Roots ──→ A ──→ B ──→ C
```

并发标记期间：

```
     GC 线程                          用户线程
       │                                │
  开始标记 A                          在运行
  A 标记完成 ✓                        │
       │                          A.field = null
       │                          （A 不再指向 B 了）
  GC 线程不知道这件事                  │
  继续标记 B                          │
  B 标记完成 ✓                        │
       │                          B.field = null
       │                          （B 不再指向 C 了）
  GC 线程不知道这件事                  │
  继续标记 C                          │
  C 标记完成 ✓                        │
       │                                │
  并发标记结束                          │
```

### 最终结果

```
A → 活的（GC 标记了）
B → 活的（GC 标记了）
C → 活的（GC 标记了）
```

但实际上 A、B、C 全是垃圾，但 GC 全部标记为存活 → **浮动垃圾**

### 为什么 GC 不知道

因为 GC 线程和用户线程是同时跑的：

- GC 线程标记 A 时，A→B 还存在
- 标记完 A 之后，用户线程才断开引用
- GC 线程已经错过了这个变化

等 GC 线程走到 B 的时候，它不知道 B 已经不可达了，还是把 B 标记为存活。

### 浮动垃圾 = 并发标记结束后才发生的引用断开

```
        SATB 队列能处理的        SATB 队列处理不了的
        ├─────────────────────┤├──────────────────┤
  ──────┼─────────────────────┼──────────────────┼──
        并发标记开始           并发标记结束         下次GC
                   ↑                           ↑
              引用断开被捕获              引用断开变成浮动垃圾
```

### 为什么不会误杀

并发 GC 的设计原则：**宁可多留垃圾（浮动垃圾），绝不误杀活对象。**

| 场景 | 会误杀吗 | 原因 |
|------|---------|------|
| 并发标记期间重新引用 | ❌ 不会 | 重新标记阶段会处理 |
| 重新标记后才重新引用 | ❌ 不会 | 重新标记是 STW，清除前引用已确定 |
| 并发清除后下次 GC | ❌ 不会 | 下次 GC 会重新检查引用关系 |

## 四、写屏障（Write Barrier）是什么

**JVM 在每次"对象引用赋值"时自动插入的一段检查代码。**

### 没有写屏障的世界

```java
// 你写的代码
A.field = B;
// JVM 执行：把 A.field 的值改成 B 的地址。完事了。
```

### 有写屏障的世界

```java
// 你写的代码
A.field = B;

// JVM 实际执行的（伪代码）
{
    Object oldRef = A.field;       // 1. 取出旧值
    A.field = B;                   // 2. 正常赋值
    writeBarrier(A, oldRef, B);    // 3. 自动插入一段代码 ← 这就是写屏障
}
```

**第 3 步是 JVM 自动加的，不是你写的。** 每次引用赋值都会触发。

### 写屏障做了什么

写屏障本身不是固定逻辑，而是一个可配置的钩子，不同收集器塞进去的代码不同：

| 收集器 | 写屏障触发时做什么 |
|--------|-------------------|
| **CMS（Incremental Update）** | 把新引用 `B` 压入 markStack |
| **G1（SATB）** | 把旧引用 `oldRef` 压入 satbQueue |
| **ZGC** | 检查染色指针状态位，必要时修正引用（自愈） |

### 读屏障 vs 写屏障

| | 写屏障 | 读屏障 |
|--|--------|--------|
| 触发时机 | `A.field = B`（赋值时） | `obj = A.field`（读取时） |
| 比喻 | 装修报备 | 入住检查 |
| 谁用 | G1、CMS | ZGC |

ZGC 的核心就是读屏障 — 每次读取对象引用时，检查染色位，发现状态不对就自动修正，这就是"自愈"。

## 五、SATB vs Incremental Update

### 两者解决同一个问题

**引用断开时怎么处理。** 两种策略完全相反。

### SATB（Snapshot-At-The-Beginning）— G1 使用

**核心思想：记录旧值。引用变化前，把旧的引用保存下来。**

```
// 伪代码
Object oldRef = this.field;      // 旧引用
this.field = newRef;              // 覆写
writeBarrier.log(oldRef);         // SATB 队列记录旧引用
```

后续对 SATB 队列中的旧引用做可达性分析 — 保证"如果引用在标记开始时可达，就不漏标"。

- **优点**：简单高效，每个引用变动只记录一次，开销固定
- **缺点**：可能多标（浮动垃圾），但不会漏标
- **重标记开销**：小（只扫队列）

### Incremental Update（增量更新）— CMS 使用

**核心思想：记录新值。引用变化后，把新的引用加入重标记集合。**

写屏障在**引用覆写之后**触发：

```java
void fieldWrite(Object newValue) {
    markStack.push(newValue);  // 把新值压入标记栈
}
```

重新标记阶段需要从 markStack 中的对象开始，重新扫描整个引用链，确认有没有因为引用变动导致漏标。

- **优点**：不会多标，垃圾清理更彻底
- **缺点**：重新标记阶段需要重新扫描某些对象的引用图，停顿时间更长、更不可预测
- **重标记开销**：大（需重扫描）

### 对比总结

| | SATB | Incremental Update |
|--|------|-------------------|
| 记录时机 | 引用覆写**前** | 引用覆写**后** |
| 记录内容 | 旧引用 | 新引用 |
| 实现者 | G1 | CMS |
| 重新标记开销 | 小（只扫队列） | 大（需重扫描） |
| 结果 | 可能多标（浮动垃圾） | 不会多标，但重标记更慢 |

### 为什么 G1 选了 SATB

G1 的目标是 `MaxGCPauseMillis=200ms` 可预测停顿。如果重标记阶段开销大（Incremental Update），停顿时间就不可控。SATB 重标记开销小且固定，所以 G1 选了它。

代价就是浮动垃圾多一点，但 G1 通过 Mixed GC 频繁回收老年代 Region 来消化这些浮动垃圾，问题不大。

## 六、新生代回收方式

**新生代统一用复制算法（Copying），区别只在于堆的组织方式不同。**

### 复制算法的原理

新生代分三个区：Eden + Survivor0 + Survivor1。对象出生在 Eden，GC 时：

1. 把 Eden + Survivor0 中**存活**的对象复制到 Survivor1
2. 清掉 Eden + Survivor0（整块回收，不需要一个个标记垃圾）
3. 下次 GC 时 Eden + Survivor1 复制到 Survivor0，来回切换

**为什么快**：新生代存活率通常不到 5%，复制 5% 的对象 + 清掉整块内存，远比逐个标记垃圾高效。

### 不同收集器的新生代处理

| 收集器 | 堆组织 | 回收方式 |
|--------|--------|----------|
| **Parallel Scavenge** | 连续物理内存 | Eden/Survivor 是堆上连续的两块区域，GC 时所有工作线程**并行**复制存活对象，全部 STW |
| **G1** | Region 化 | Eden/Survivor 是标记为特定类型的 Region，GC 时从垃圾最多的 Eden Region 复制存活对象到新的 Survivor Region |
| **CMS** | 不负责新生代 | 新生代交给其他收集器（如 Parallel Scavenge），CMS 只管老年代 |

**核心不变**：都是复制存活对象 → 清掉整块区域。

## 七、老年代回收策略对比

**新生代一律复制算法，老年代看收集器：**

| 收集器 | 老年代算法 | 特点 |
|--------|-----------|------|
| CMS | **标记-清除** | 有碎片 |
| G1 | **标记-复制 + 标记-整理** | Region 间复制存活对象，无碎片 |
| ZGC | **染色指针 + 并发整理** | 本质上也是复制，无碎片 |

### CMS 为什么主要用于老年代

老年代对象存活率高，用复制算法浪费内存。CMS 用标记-清除，不需要复制存活对象，但会产生内存碎片。

新生代存活率极低（通常 <5%），复制算法更高效，CMS 的标记-清除在新生代没有优势。

## 八、G1 为什么大堆才划算（> 6GB）

### G1 的额外开销

- **Region 元数据**：堆切成上千个 Region，每个 Region 需要状态跟踪（Eden/Survivor/Old/Free）
- **RSet（Remembered Set）**：每个 Region 维护一个 RSet，记录"哪些其他 Region 引用了自己"。RSet 写入要在每次引用赋值时触发写屏障，RSet 的大小通常占堆的 5%-20%
- **Collection Set 筛选**：每次回收前要评估哪些 Region 垃圾最多，有额外 CPU 开销
- **并发标记开销**：G1 的并发标记+SATB 写屏障在小堆上收益不大（老年代本身不大，标记很快就完成了）

### 堆大小 vs 收集器选择

| 堆大小 | G1 | Parallel/CMS |
|--------|-----|-------------|
| < 2GB | RSet 占比高，内存浪费 | 简单分代，内存利用率高，GC 快 |
| 2-4GB | 收益有限 | Parallel 的 Full GC 也很短 |
| 4-8GB | 开始有优势 | CMS 碎片问题开始暴露 |
| > 8GB | 优势明显：分区回收避免整堆扫描 | Full GC 停顿急剧增长 |

**简单说**：G1 的优势是"把大堆切成小块，每次只回收最有价值的块"。但堆小的时候，整堆扫描本身就很快，不需要切块。切块反而带来 RSet 和元数据的固定开销。

## 九、ZGC 的多重映射（Multi-Mapping）

### 为什么需要这个

ZGC 的核心技术是**染色指针（Colored Pointer）**：在 64 位指针的高位（第 42-45 位）塞 4 个状态位：

```
63      46 45 44 43 42 41        0
[ unused  | M0 M1 Remapped Marked0 | 对象偏移 ]
```

这要求指针本身携带元数据，但正常情况下 OS 只允许每个物理页映射到一个虚拟地址。

### 多重映射的实现

ZGC 在启动时对同一块物理内存建立 3-4 个虚拟地址映射：

```
物理页 P：
  虚拟地址 0x0000_1000_0000_0000  → P  (标记为 Marked0)
  虚拟地址 0x0000_2000_0000_0000  → P  (标记为 Marked1)
  虚拟地址 0x0000_3000_0000_0000  → P  (标记为 Remapped)
```

同一个对象，在不同的染色位下，指针值不同（高位状态位不同），但指向的是同一块物理内存。

### 解决了什么

1. **指针着色和实际寻址分离** — 指针高位存状态位不影响寻址，三个虚拟地址都能正确访问到同一物理页
2. **读屏障的实现基础** — ZGC 的读屏障在访问对象时检查染色位，如果发现状态位不对，自动通过自愈（self-healing）把指针修正到正确的虚拟地址
3. **并发整理的基础** — ZGC 可以并发地搬移对象，同时旧指针通过 Remapped 虚拟地址仍然能访问到正确数据

### 一句话总结

多重映射 = 同一块物理内存，多个虚拟地址，每个地址对应不同的染色状态。ZGC 通过这种方式让 64 位指针既当地址又当状态标记，实现了几乎零停顿的并发垃圾回收。

## 十、概念之间的逻辑链

```
GC Roots 是可达性分析的基础
    │
    ▼
Concurrent Mark 并发标记给存活对象画勾
    │
    ▼
并发阶段用户线程修改引用 → 浮动垃圾
    │
    ▼
SATB / Incremental Update 是解决引用变动的两种策略
    │
    ▼
G1 用 SATB 保证停顿可控，但有固定开销（小堆不划算）
    │
    ▼
ZGC 用染色指针+读屏障+多重映射把停顿压到更极致
```

## 复习骨架

- GC Roots = 可达性分析起点（栈/静态变量/常量/JNI/锁）
- 并发标记 = 给存活对象画勾，标记位存在对象头
- 浮动垃圾 = 并发标记结束后用户线程断开引用，这次 GC 收不掉
- 写屏障 = JVM 在引用赋值时自动插入的钩子代码
- SATB = 记录旧引用（G1），重标记开销小，可能有浮动垃圾
- Incremental Update = 记录新引用（CMS），重标记开销大，不会多标
- 新生代一律复制算法，老年代看收集器（CMS=标记-清除，G1/ZGC=标记-复制）
- G1 大堆才划算：RSet + Region 元数据开销在小堆上不值得
- ZGC 多重映射：同一物理页多个虚拟地址，染色指针实现并发整理
