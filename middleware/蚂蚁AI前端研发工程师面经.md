## 一面（技术基础）

### 1. 事件循环机制 ★★

**考点**: JS 运行机制

**答案要点**:
- 事件循环是 JS 处理异步任务的机制，通过不断循环执行栈和任务队列来调度任务
- 执行栈为空时，先清空微任务队列
- 接着执行一个宏任务
- 再次检查微任务队列，循环往复
- 微任务包括 Promise.then、MutationObserver 等
- 宏任务包括 setTimeout、setInterval、I/O 等

**常见坑**:
- 混淆微任务和宏任务的优先级
- 忽略了 UI 渲染在事件循环中的位置

**追问**: 如果在微任务中不断产生新的微任务，会发生什么？

### 2. 手写防抖函数 ★★

**考点**: 手写代码、应用场景

**答案要点**:
- 防抖是在事件触发后延迟一段时间执行，若时间内再次触发则重新计时
- 实现逻辑：利用闭包保存定时器 ID，每次触发清除旧定时器并重置
- AI 场景：用于搜索建议或实时 Prompt 补全，减少对后端 API 的请求频率
- 关键点：支持立即执行参数，支持取消功能

**常见坑**:
- 忘记清除定时器导致逻辑错误
- 未处理 this 指向和参数传递

**追问**: 如何实现一个支持立即执行的防抖函数？

### 3. Promise ★

**考点**: 异步编程

**答案要点**:
- Promise 是异步编程的一种解决方案，通过对象状态管理异步操作，解决了回调地狱问题
- 三种状态：Pending、Fulfilled、Rejected
- 链式调用：通过 then 返回新 Promise 实现
- 错误处理：统一的 catch 捕获异常

**常见坑**:
- 不理解 Promise 的状态一旦改变就不可逆
- 对 Promise.all 和 Promise.race 的使用场景混淆

**追问**: 如果有多个并发请求，如何保证它们按顺序返回结果？

### 4. 流式输出（Streaming）★★

**考点**: AI 前端基础、Fetch API

**答案要点**:
- 流式输出是指服务端将数据分片传输，前端通过 ReadableStream 实时接收并渲染
- 技术实现：使用 Fetch API 的 Response.body 获取 ReadableStream
- 处理方式：通过 TextDecoder 解码数据块，逐步更新 UI
- 优势：减少首字等待时间，适合 AI 对话场景

**常见坑**:
- 忽略了流式数据中可能包含的特殊字符或不完整 JSON
- 未处理连接中断的重连逻辑

**追问**: 如果流式输出中途报错，前端应该如何优雅地展示？

### 5. CSS 盒模型 ★

**考点**: CSS 基础

**答案要点**:
- 标准盒模型：width = content
- IE 盒模型：width = content + padding + border
- box-sizing 属性：用于切换两种模型

**常见坑**: 混淆 margin 的合并规则；对 display: inline-block 的间隙问题不了解

**追问**: 如何实现一个垂直居中的布局？

### 6. 闭包 ★★

**考点**: JS 作用域

**答案要点**:
- 闭包是指有权访问另一个函数作用域中变量的函数，本质是函数与其词法环境的引用捆绑
- 形成条件：函数嵌套 + 内部函数引用外部函数的变量
- 常见应用：封装私有变量、柯里化、防抖节流、模块化模式
- 内存影响：闭包会使被引用的变量无法被 GC 回收，需注意内存泄漏

**常见坑**:
- 认为闭包一定会导致内存泄漏
- 不理解闭包在循环中引用变量的陷阱

## 二面（深入原理与项目）

### 1. Markdown 渲染问题 ★★

**考点**: 项目实践、性能优化

**答案要点**:
- 解析器选择：如 marked 或 react-markdown
- 增量处理：通过缓冲区合并数据，避免频繁触发 DOM 重绘
- 代码高亮：集成 Prism.js 或 Highlight.js
- 安全性：使用 DOMPurify 防止 XSS 攻击

**常见坑**:
- 频繁触发全量渲染导致页面卡顿
- 未处理流式输出中 Markdown 语法被截断的情况

**追问**: 如果 Markdown 内容非常长，如何优化渲染性能？

### 2. React Fiber 架构 ★★★

**考点**: 框架原理

**答案要点**:
- 将渲染过程拆分为可中断的单元，通过时间分片机制避免主线程阻塞
- 核心原理：将同步递归渲染改为异步可中断的链表遍历
- 优先级调度：根据任务紧急程度分配执行时间
- 增量渲染：在空闲时间执行任务，保证 UI 响应

**常见坑**:
- 混淆了 Fiber 和虚拟 DOM 的概念
- 不清楚 React 18 的并发特性

### 3. RAG 与前端角色 ★★★

**考点**: AI 架构

**答案要点**:
- RAG 通过检索外部知识库来增强 LLM 的回答准确性，前端负责交互和上下文呈现
- 前端职责：构建检索查询界面、展示引用来源、处理多模态数据
- 关键点：向量化搜索的 UI 交互、上下文窗口管理
- 优化方向：减少 Token 消耗，优化检索结果的展示形式

**常见坑**:
- 忽略了前端对检索结果的过滤和排序
- 未考虑用户对检索结果的反馈机制

### 4. 前端性能监控 ★★

**考点**: 性能优化

**答案要点**:
- 关键指标：LCP、FID、CLS
- 采集方式：使用 Performance API 和 PerformanceObserver
- 上报策略：批量上报、采样上报、异常捕获

**常见坑**:
- 监控脚本本身影响了页面性能
- 忽略了不同网络环境下的数据差异

**追问**: 如何定位一个线上偶发的白屏问题？

### 5. 前端模块化 ★★

**考点**: 工程化

**答案要点**:
- 加载机制：CJS 是运行时同步加载，ESM 是编译时异步加载
- 导出方式：CJS 导出的是值的拷贝，ESM 导出的是值的引用
- 静态分析：ESM 支持 Tree Shaking，有利于打包优化

**常见坑**: 混淆 require 和 import 的执行时机；对循环依赖的处理方式不清楚

### 6. 跨域 ★

**考点**: 网络协议

**答案要点**:
- 常见方案：CORS（服务端设置）、代理服务器（Nginx/Webpack Dev Server）、PostMessage
- AI 场景：通常通过后端代理转发 API 请求，避免暴露 API Key

**常见坑**: 误以为 JSONP 是现代主流方案；不了解 CORS 预检请求（OPTIONS）的机制

### 7. 原型链 ★★

**考点**: JS 核心原理

**答案要点**:
- 原型链是对象查找属性的机制，通过 `__proto__` 指向原型对象，实现继承
- 核心：每个对象都有原型，原型也是对象，直到指向 null
- 继承：通过构造函数的 prototype 属性实现共享方法
- 意义：节省内存，实现对象间的逻辑复用

**常见坑**: 混淆 prototype 和 `__proto__`；不理解 Object.create 的原理

**追问**: 如何实现一个寄生组合式继承？

### 8. 长对话上下文管理 ★★★

**考点**: AI 应用开发

**答案要点**:
- 长对话受限于模型上下文窗口，需要前端配合后端进行上下文截断或摘要
- 策略：滑动窗口、摘要压缩、关键信息提取

## 三面（架构与综合能力）

### 1. 设计 AI Agent 前端工作台 ★★★

**考点**: 系统设计

**答案要点**:
- 核心模块：对话交互区、Agent 状态监控（思维链）、工具执行面板、配置管理
- 关键技术：WebSocket 实时通信、复杂状态管理（如 Zustand/Redux）、组件化编排
- 交互设计：支持人工介入（Human-in-the-loop）的确认机制

**常见坑**:
- 忽略了 Agent 任务执行过程中的错误处理和重试机制
- 界面过于复杂导致用户认知负担过重

**追问**: 如何设计一个通用的工具调用协议，让 Agent 能调用前端定义的函数？

### 2. 项目最大挑战 ★★★

**考点**: 项目复盘、解决问题能力

**答案要点**:
- 通过 STAR 原则（情境、任务、行动、结果）描述一个具有技术深度的案例
- 难点：比如复杂的性能瓶颈、跨团队协作、技术选型冲突
- 行动：调研、方案对比、落地实施、复盘总结
- 结果：量化的提升指标（如性能提升 30%，开发效率提升 20%）

**常见坑**: 描述过于平淡，体现不出技术深度；把功劳全揽在自己身上，忽略团队协作

### 3. AI 对前端开发的影响 ★★

**考点**: 技术视野

**答案要点**:
- 影响：前端从 UI 开发转向 AI 应用开发，更关注交互逻辑和数据流
- 工具使用：Copilot 辅助编码、ChatGPT 进行架构设计、AI 调试工具
- 观点：AI 是助手，核心竞争力在于对业务的理解和架构设计能力

**常见坑**: 过度依赖 AI 导致基础能力退化；对 AI 的局限性缺乏认知

### 4. 新技术选型分歧处理 ★★

**考点**: 团队协作

**答案要点**:
- 评估：对比方案的优劣、风险、成本、可维护性
- 沟通：组织技术评审，听取不同意见，进行小范围验证（POC）
- 决策：尊重团队共识，若无法达成则由负责人拍板，并记录决策过程

**常见坑**: 情绪化争论，不关注技术本质；忽略了团队现有的技术栈和学习成本
